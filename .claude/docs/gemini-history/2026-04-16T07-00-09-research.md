# 2026 AI 코딩 하네스 구조 및 자기발전 시스템 트렌드 분석 보고서

## 요약
- 2026년 AI 코딩 하네스는 단순 챗-프롬프팅을 넘어 'Plan-Execute-Verify' 루프와 자율 에이전트 스웜(Swarm)을 결합한 멀티-에이전트 오케스트레이션 구조로 진화했습니다.
- 멀티 모델 라우팅은 물리적 코드량(예: 30줄) 기준을 탈피하여, 작업 의도(Intent)와 메타데이터를 기반으로 하는 시맨틱 라우팅 및 병렬 서브 에이전트 패턴이 표준이 되었습니다.
- 실패 패턴을 단순 JSON에 기록하는 것을 넘어, AI가 스스로 새로운 스킬을 정의하고 코드를 수정하는 '메타인지' 기반 자기 발전(Self-improving) 구조가 핵심 경쟁력으로 자리 잡았습니다.

## 시사점: 우리 v6.5 구조 개선 아이디어

1. **[우선순위 1] 하드코딩된 라우팅 개선 (Semantic Routing 적용)**
   - **아이디어:** 현재 'Codex=30줄 이상, Claude=30줄 이하' 등 코드량 기준의 라우팅을 폐기하고, 경량 모델을 프록시로 두어 '의도(Intent)'와 '필요 컨텍스트 량'에 따라 동적으로 라우팅하는 스크립트(`on-prompt-unified.mjs` 고도화) 도입.
   - **트레이드오프:** 라우팅 판단을 위한 1차 LLM 호출 비용 및 미세한 지연 시간 발생.

2. **[우선순위 1] Git Worktree 기반 병렬 서브 에이전트 실행**
   - **아이디어:** CTO(Opus)가 동적 팀(스킬 템플릿) 편성 시, 각 에이전트가 백그라운드 Git Worktree에서 독립적으로 코드를 작성한 후 머지하는 훅 추가. 다중 파일 수정 속도 대폭 향상.
   - **트레이드오프:** 로컬 스토리지 점유율 증가 및 충돌 시 머지를 해결할 'Healer' 에이전트 추가 구현 필요.

3. **[우선순위 2] `evolving-rules.json`의 메타인지(Self-improving)화**
   - **아이디어:** 현재 단순 텍스트 누적을 넘어, 특정 에러(예: 의존성 에러)가 반복될 경우 AI가 능동적으로 `.claude/agents/memory/` 경로에 새로운 대응 스킬(`.SKILL.md`)을 작성하고 로드하는 '자율 스킬 생성 훅' 도입.
   - **트레이드오프:** 잘못된 규칙이 스킬로 생성될 수 있으므로, 생성된 스킬에 대한 휴먼 승인(Post-review) 단계가 필요할 것으로 추정됨.

4. **[우선순위 2] 정적 7개 에이전트의 완전 동적화 (Dynamic Swarm)**
   - **아이디어:** 현재 7개의 코어 에이전트 파일을 유지보수하는 대신, CTO(Opus)가 태스크에 맞춰 'pm/qa' 등의 역할을 즉석에서 인스턴스화하고 작업 완료 후 폐기하는 메타 오케스트레이터로 변경.
   - **트레이드오프:** 프롬프트 관리가 분산에서 중앙(CTO)으로 집중되어 CTO 프롬프트 튜닝 난이도가 높아짐.

5. **[우선순위 3] MCP를 활용한 2단 구조(hub-특화) 컨텍스트 최적화**
   - **아이디어:** 현재 working(hub)과 각 프로젝트로 나뉜 구조에서 복사/동기화를 하는 대신, MCP 서버를 띄워 CTO 모델이 각 프로젝트의 컨텍스트를 DB 조회하듯 필요한 만큼만 동적으로 당겨오게 설정.
   - **트레이드오프:** MCP 서버 구축 및 유지보수 리소스 소요.

6. **[우선순위 3] 토큰 최적화 프록시 훅(RTK 스타일) 적용**
   - **아이디어:** 현재 터미널 도구의 방대한 출력을 그대로 AI에 던지는 구조에서, 정규식과 휴리스틱을 사용해 무의미한 터미널 로그를 필터링하는 토큰 커터(Token Cutter) 훅 추가.
   - **트레이드오프:** 중요한 에러 로그가 우발적으로 필터링될 위험.

7. **[우선순위 3] 백그라운드 데몬 및 모바일 Handoff 세션 지원**
   - **아이디어:** CLI 기반의 동기식 작업 외에 긴 리서치나 대규모 리팩토링은 `nohup` 스타일로 백그라운드(`post-push-pending.md` 연동)에 위임하고, 결과를 태블릿에서 확인할 수 있도록 웹훅 연동.
   - **트레이드오프:** 세션 상태 관리(Locking) 등 상태 동기화 아키텍처 추가 구현이 필수적임.

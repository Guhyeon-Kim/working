# Success Patterns

> 성공한 접근법과 패턴 기록. 동일한 성공을 반복하기 위한 양방향 학습 시스템.
> (failure-cases.md = "하지 말 것" / success-patterns.md = "이렇게 하면 잘 됨")

---

## 기록 기준

- 파이프라인이 BLOCK 없이 한 번에 통과한 경우
- CEO가 결과물을 수정 요청 없이 승인한 경우
- 이전에 실패했던 유형의 작업이 개선된 방법으로 성공한 경우
- 예상보다 빠르거나 효율적으로 완료된 경우

---

## 성공 패턴

### [SP-001] Codex 지시어에 타입 정의 선제공

**상황**: 새 컴포넌트 구현 시
**패턴**: Codex에 코드 구현을 맡기기 전, 사용할 타입/인터페이스 정의를 지시어에 포함
**효과**: 타입 에러 재작업 감소, qa-agent 1차 통과율 상승
**적용**: 구현 단계에서 Codex 지시어 작성 시 항상

---

### [SP-002] design-guide-v2 참조 링크를 프롬프트에 명시

**상황**: UI 관련 작업 (디자인, 프론트엔드 구현)
**패턴**: Gemini/Codex 프롬프트에 "design-guide-v2.md Section N 참조" 구체적 섹션 지정
**효과**: 디자인 가이드 위반 재작업 0건 (vs 미참조 시 평균 2~3건 수정)
**적용**: UI 관련 작업 시 항상

---

### [SP-003] 5단계 파이프라인 순차 준수

**상황**: 신규 기능 개발
**패턴**: 기획 승인(CEO 확인①) 없이 설계·구현에 착수하지 않음
**효과**: "이거 아닌데" 피드백으로 인한 전면 재작업 방지
**적용**: 모든 신규 기능 (긴급 버그 제외)

---

### [SP-004] git stash/브랜치 분기 후 대규모 변경

**상황**: 10+ 파일 변경 예상 작업
**패턴**: 작업 전 `git stash` 또는 별도 브랜치 분기 → 실패 시 즉시 복원 가능
**효과**: search/page.tsx 파손 사건(2026-03-18) 이후 동류 장애 0건
**적용**: 대규모 변경, 디자인 전면 개편(REDESIGN-*) 시

---

### [SP-005] 검증 에이전트 병렬 실행 + 모델 최적화

**상황**: 4단계 검증
**패턴**: pr-review-toolkit 5개 에이전트를 Sonnet 모델로 병렬 실행, qa-agent만 Opus
**효과**: 검증 품질 유지 + 토큰 절감 (Opus 5개 → Sonnet 5개 + Opus 1개)
**적용**: 중규모 이상 변경의 검증 단계

## 기록 형식

```
### [SP-NNN] 패턴명

**상황**: 어떤 작업 유형에서 발생
**패턴**: 구체적으로 무엇을 했는가
**효과**: 측정 가능한 결과 (재작업 횟수, 통과율, 시간 등)
**적용**: 언제 이 패턴을 사용해야 하는가
```

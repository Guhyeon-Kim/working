## 2. 핵심 결정

### 2-1. 에이전트 구조 전환 (A → B)

- **기존 A**: 15 에이전트 + 5단계 고정 파이프라인 (pm → research → design →
  data → front/back → qa → security → infra)
- **신규 B**: 7 공통 에이전트 + CTO 판단 기반 동적 편성
- **이유**:
  - A는 경량 작업(블레벨 타로 카드 1개 추가)에도 전 단계 통과 요구 → 우회됨
  - B는 최소 팀(Claude + Codex)이 기본, 필요 시에만 확장
  - CTO가 매 작업 시 팀 구성을 "선언"하므로 편성 근거가 학습 신호로 축적됨

### 2-2. 얇은 에이전트 + 두꺼운 스킬/훅

- 에이전트 = "누가 지시하느냐"의 얇은 껍질 (프롬프트 + 주 CLI)
- 실제 능력 = 스킬·훅·MCP 공용 자산
- 에이전트는 7개로 고정. 성장시키지 않음
- 스킬·훅·MCP가 성장 포인트 (고도화로 시스템 똑똑해짐)
- **이유**: 에이전트 추가는 복잡도 증가, 스킬·훅은 합산 효과

### 2-3. docs 폴더 3분할

```
docs/
├── current/  # 활성 문서 (버전 관리)
├── asis/     # 이전 메이저 버전 스냅샷
└── logs/     # 날짜 기반 보조 로그 (Notion이 primary)
```

- **이유**: 로그와 문서는 성격이 다름. 섞으면 버전 업 시점 관리 어려움.

### 2-4. 양식 버전업 시 과거 기록 보존 (방식 1)

- 양식 v1.0으로 작성된 과거 기록은 **그대로 유지** (리트로핏 금지)
- v2.0부터 새 포맷 적용
- 각 기록 상단 `양식버전` 필드로 구분
- **이유**: "과거의 것을 모두 업데이트할 순 없다. 그렇게 발전하고 성장하는 것."
  (CEO 발언, 2026-04-22)

### 2-5. devlog 저장은 Notion DB 중심

- **Primary**: Notion 💻 개발 노트 (스키마 v2.0)
- 파일 로그(`docs/logs/`)는 보완 수단
- **이유**:
  - UI로 집계·필터·차트 가능 → "30일치 있음" 체감 확보 (CEO의 헛헛함 해소)
  - MCP로 Claude가 양방향 접근
  - 모바일에서도 확인 가능

### 2-6. 키 관리 Notion 중앙화

- **Primary**: Notion 🔑 서비스/환경변수 목록
- 값은 페이지 **본문 코드블록**에 실제 값 그대로 (마스킹·편집 금지)
- Just-in-time 등록: 일괄 이관 불가능 (CEO가 현재 키값을 모름)
- **이유**:
  - 기존에 키가 로컬 .env / Vercel / GitHub Secrets / 기억 분산 → 관리 비용 폭증
  - "어느 키가 어디 있는지"부터 추적해야 하는 상태
  - Notion은 3rd-party API 키(결제 secret 아님) 레벨에서 보안 허용 범위

### 2-7. CEO/CTO 역할 명문화

- CEO = 사용자. 아이디어 전달, 방향성 결정, 최종 승인, 리스크 수용
- CTO = Claude. 요구사항 구조화, 팀 편성, 품질 책임, 작동 증거 수집
- CTO는 예스맨 아님. 범위 폭주·과거 모순 요청에 이의 제기가 본업
- **이유**: 솔로 개발자의 구조적 한계. "이거 진짜 돌아요?" 물어주는 동료 역할.

### 2-8. MVH (Minimum Viable Harness) 원칙

- 완벽하지 않은 채 돌아가는 것 > 완벽히 설계됐으나 안 돌아가는 것
- 이번 구축 범위 엄격히 제한 (§3 참조)
- 보류 사항은 §4에 기록 후 2일 검증 통과 후 재산정
- **이유**: v5.3의 껍데기화가 "완벽 설계" 욕심에서 비롯됨

### 2-9. 선택과 집중 — 3 focus로 축소

- **기존**: 8개 워크스트림 병행 (회사 본업, 허브와이즈, 블레벨, 자동화, 경진대회 3건, 하네스)
- **신규**: active 3개 (허브와이즈 / 컨텐츠 자동화 / 하네스), 블레벨은 검증 후 release candidate 판단
- **이유**:
  - Day 1 세 번째 프로젝트 리뷰에서 드러난 공통 패턴: "많이 만들었는데 뾰족하지 않다"
  - 8 워크스트림은 한 사람의 주의력 용량 초과 → 모든 결과물이 '미완' 상태
  - 하네스가 해결할 수 없는 문제. 하네스 강화가 아니라 **선택**이 해결책
- **적용**: CLAUDE.md §1-0에 명문화. CTO는 focus 추가 요청에 이의 제기 의무

### 2-10. 작업 환경 분리 (Mode A/B/C)

- **배경**: 집(제약 없음) vs 회사(네트워크·정책 제약) 두 환경을 단일 워크플로로 통합하려는 시도가 스트레스와 Codespaces 한도 소진의 근본 원인
- **결정**: 환경 통합 대신 **작업을 모드로 분리**
  - Mode A (깊은 구현) = 집에서만
  - Mode B (기획·사고) = 회사 기본값
  - Mode C (경량 수정) = 어디서든
- **적용**: CLAUDE.md §1-5에 명문화. Codespaces 운영 원칙(§2-7)과 연계
- **부수 효과**: 기획자 본연의 리듬(낮=사고, 저녁=구현)과 자연스럽게 정합

## 4. 보류 사항 (Phase 2 후보)

2일 검증 후 우선순위 재산정. 현재 MVH 범위 제외.

### 4-1. 세션 훅 자동화

**필요성**: 현재는 "세션 시작 시 CLAUDE.md 로드" / "종료 시 devlog 작성"을
Claude가 명시적 수행. 훅 자동화 필요.

**구현 안**:
- `hooks/session-start.mjs`: settings.json의 SessionStart에 걸어두고
  CLAUDE.md + docs/current/ 5개 자동 로드
- `hooks/session-end.mjs`: Stop 훅에 걸어두고 Notion devlog 등록 트리거
- 검증 로그: `~/.claude/hook-execution.log`에 타임스탬프 append

**보류 사유**: 훅 연결 재검증 필요. 이전에 "등록했는데 안 불러진" 사례 재발 방지
위해 검증 로그부터 먼저 확인.

### 4-2. `scripts/delegate.mjs` 재작성

**필요성**: 기존 delegate.mjs는 15개 에이전트명을 알고 있음. 7개로 축소되면서
매핑 깨짐.

**구현 안**:
```js
const AGENTS = {
  researcher: { cli: 'gemini', model: 'gemini-2.5-pro' },
  planner: { cli: 'claude', model: 'opus' },
  copywriter: { cli: 'gemini', fallback: 'claude' },
  designer: { cli: 'claude', mcp: ['figma'] },
  builder: { cli: 'codex' },
  tester: { cli: 'codex', mcp: ['playwright'] },
  curator: { cli: 'claude' },
};
```

**보류 사유**: 기존 스크립트가 현재 작동 중이므로 급하지 않음. 7개 에이전트
스텁이 실제 호출 테스트될 때 함께 업데이트.

### 4-3. evolving-rules 재설계

**진단 결과** (어제 분석):
- 기존 감지 로직은 10개 고정 키워드만 매칭 (typescript-error, async-params 등)
- 실제 실패의 70%는 "요구사항 오해", "설계 결함" 같은 카테고리 → 감지 대상 아님
- 임계값 `count ≥ 5`가 너무 높음 (실제 발동 0건)
- 성공 패턴 캡처 장치 부재 → 실패만 반응적으로 잡는 CCTV

**재설계 방향**:
- 감지 소스 확장: CEO 컨펌 반려 로그 + 명시적 세션 회고
- 임계값 완화: count ≥ 1 → candidate-patterns.md (관찰 기록)
- 성공 패턴 캡처: 1회 한 번에 통과한 워크플로도 기록
- Recall 검증: session-start에서 규칙 로딩 → 실제 참조 여부 확인

**보류 사유**: 인프라(훅 + 영속화) 재정비 후 진행해야 의미 있음.

### 4-4. skills/ 재분류

**필요성**: 19개 스킬이 v5.3 에이전트 기준으로 분류됨. 7개로 줄면서 매핑 재검토.

**구현 안**: 각 스킬에 "권장 에이전트" 메타데이터 추가 (스킬 파일 상단 frontmatter).

**보류 사유**: 실사용 중 혼란 없으면 기존 분류 유지 가능.

### 4-5. hooks/ 정리

**필요성**:
- 레거시 `.sh` 파일 (Windows cmd 미지원)
- 중복된 on-prompt / on-stop 구버전
- `_auto-heal.mjs` v6.4 검증 필요

**보류 사유**: 건드리면 연쇄 검증 필요. 이번 범위 밖.

### 4-6. `.gitignore` 재검토

**필요성**: 현재 `.claude/agents/memory/` 전체 제외로 evolving-rules 영속 불가.
운영 결정 후 일부 경로 해제 필요.

**구현 안 후보**:
- 옵션 A: 전체 해제 → git 이력 지저분
- 옵션 B: Notion으로 export 후 memory/ 비우기 → 복잡
- 옵션 C: `.claude/agents/memory/important/`만 예외 처리 → 절충

**보류 사유**: 옵션 결정 전 Notion 활용도 2일 관찰.

### 4-7. 프로젝트 레포 patch 전파

**필요성**: working의 v6.0 하네스를 허브와이즈·B무거나·컨텐츠 자동화 레포로 적용.

**구현 안**: `/project:patch` 슬래시 커맨드로 각 레포에 동기화.

**보류 사유**: working 자체 검증 통과 후. 프로젝트 레포가 먼저 망가지면 복구 비용 큼.

### 4-8. 월 1회 성장 리포트 자동 생성

**필요성**: "이번 달 몇 건 학습했나" 자동 집계. CEO의 헛헛함 해소 직접 장치.

**구현 안**: Notion MCP로 💻 개발 노트 집계 → 월간 페이지 자동 생성.

**보류 사유**: 데이터 누적 없으면 리포트 의미 없음. 최소 2주 운영 후.

### 4-9. `on-stop.mjs` `session.started` epoch 0 버그

contents-auto 실측에서 `1970-01-01T00:33:46.000Z` 계산 확인.

재현·수정 필요 (2026-04-22 contents-auto 통합 중 발견).

### 4-9. 키 관리 .env 자동 주입

**필요성**: Notion 🔑 DB에서 조건부(프로젝트·환경) 쿼리 → `.env.local` 생성.

**구현 안**:

또는 Claude Code 내에서 MCP로 직접 조회 후 파일 생성.

**보류 사유**: 키 이관이 진행 중 (Just-in-time). 이관 완료 후 구현.

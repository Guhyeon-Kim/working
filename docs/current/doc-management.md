# 문서 관리 지침 (Document Management)

**양식**: doc-management v1.0
**작성일**: 2026-04-22

---

## 0. 이 문서의 목적

working 레포의 모든 문서·로그·기록의 관리 방식을 정의한다. 핵심 질문:

- 문서는 어디에 두는가?
- 버전 업은 어떻게 하는가?
- 과거 기록은 어떻게 처리하는가?
- 로그는 어디에 쌓이는가?
- Notion과 파일은 어떻게 역할 분담하는가?

---

## 1. 폴더 구조

```
docs/
├── current/     # 활성 문서 (버전 관리 대상)
├── asis/        # 이전 메이저 버전 스냅샷
└── logs/        # 날짜 기반 보조 로그 (Notion이 primary)
```

### 1-1. `docs/current/`

**담는 것**: 현재 활용 중인 지침·양식·카탈로그
**특성**:
- 버전 관리 대상
- Claude는 세션 시작 시 이 폴더 전체를 로드
- 이 폴더에서 참조되지 않는 문서는 사실상 죽은 문서

**현재 포함 문서**:
- `design-decisions.md` — 설계 결정 기록
- `team-composition.md` — 팀 편성 지침
- `doc-management.md` — 이 문서
- `key-management.md` — 키 관리 지침
- `agents-catalog.md` — 에이전트 카탈로그

### 1-2. `docs/asis/`

**담는 것**: 이전 메이저 버전의 문서 스냅샷
**특성**:
- 읽기 전용 (수정 금지)
- 참고용, 현재 운영에 적용되지 않음
- 네이밍: `<원본명>-v<버전>.md`

**예시**:
- `CLAUDE-v5.3.md` — v5.3 시절 CLAUDE.md
- `team-composition-v1.0.md` (언젠가 v2.0 나오면)

**주의**:
- 모든 수정본을 asis에 쌓지 말 것 (메이저 변곡점만)
- 오타·문구 수정은 git 히스토리로 추적

### 1-3. `docs/logs/`

**담는 것**: 날짜 기반 보조 로그
**특성**:
- **Notion이 primary**, 이 폴더는 보조
- git 내부에서 작업 히스토리 확인 필요할 때 사용
- Notion 접근 불가 환경(오프라인, Notion 장애)용 백업

**네이밍**:
```
docs/logs/YYYY-MM/YYYY-MM-DD.md
```

**예시**:
- `docs/logs/2026-04/2026-04-22.md`

**사용 시점**:
- Claude가 세션 종료 시 Notion 등록과 동시에 파일로도 append
- 월별 정리는 월말에 CEO 요청 시 수행

## 2. 버전 업 2유형

버전 업에는 성격이 다른 두 종류가 있다.

### 2-1. 유형 A: 시스템/도구 버전업

**대상**:
- `CLAUDE.md`
- `docs/current/team-composition.md`
- `docs/current/agents-catalog.md`
- `agents/*.md` (에이전트 프롬프트)
- `skills/*.md` (스킬 정의)

**절차**:
1. 기존 파일을 `docs/asis/<name>-v<old>.md`로 복사
2. `docs/current/<name>.md`를 `v<new>` 내용으로 덮어쓰기
3. 새 내용에 버전 변경 이력 섹션 업데이트
4. 커밋 메시지: `bump: <name> v<old> → v<new>`

**특성**: **파일 자체가 교체**됨. 과거 상태는 asis에서만 참조.

**예시** (CLAUDE.md v5.3 → v6.0):
```bash
cp CLAUDE.md docs/asis/CLAUDE-v5.3.md
# 루트 CLAUDE.md를 v6.0 내용으로 덮어쓰기
git add CLAUDE.md docs/asis/CLAUDE-v5.3.md
git commit -m "bump: CLAUDE.md v5.3 → v6.0"
```

### 2-2. 유형 B: 양식/템플릿 버전업

**대상**:
- devlog 양식 (Notion 스키마 + 본문 포맷)
- 로그 파일 양식 (`docs/logs/YYYY-MM-DD.md`)
- 리포트 양식 (월간 리포트 등)

**절차**:
1. 양식 문서(`docs/current/<format-name>.md`)를 `docs/asis/`로 복사
2. `docs/current/<format-name>.md`를 v<new>로 업데이트
3. **과거 기록은 건드리지 않음** (리트로핏 금지)
4. 이후 작성되는 신규 기록만 새 양식 적용
5. 신규 기록의 `양식버전` 메타데이터 필드로 구분

**특성**: **양식 문서는 교체**, **과거 기록은 보존**. 기록의 시간 축을 따라가면
양식 진화 자체가 드러남.

**예시** (devlog 양식 v1.0 → v2.0):
- 2026-04-01~15: devlog v1.0 양식 (필드 4개)
- 2026-04-22: 양식 v2.0 전환 (필드 11개)
  - 기존 v1.0 엔트리 리트로핏 안 함
  - 각 엔트리의 `양식버전` 필드로 구분
- 2026-04-23~: 모든 신규 엔트리는 v2.0

## 3. devlog 작성 지침

### 3-1. 기본 원칙

- **Primary**: Notion 💻 개발 노트 DB (`collection://1212c0d1-40ed-425e-915f-eedca3577add`)
- **보조**: `docs/logs/YYYY-MM/YYYY-MM-DD.md`
- **작성 시점**: 매 세션 종료 시
- **필수 범위**: 하네스 자체 작업(δ), 프로젝트 작업(α/β/γ/custom) 모두 기록

### 3-2. v2.0 스키마 필드

| 필드 | 타입 | 작성 지침 |
|---|---|---|
| **이름** | title | 작업 타이틀. 20자 내외, 동사형. 예: "허브와이즈 편향지수 API 신규" |
| **상태** | select | 계획/진행중/완료. 완료 후에도 후속 필요하면 "진행중" 유지 |
| **카테고리** | select | 기능/버그수정/리팩토링/인프라/보안/QA/기타 |
| **담당** | select | CEO/Claude/Gemini/Codex/수동/시스템 — 주 수행자 |
| **프로젝트** | relation | 📁 프로젝트 DB 중 하나 선택 (필수) |
| **내용** | text | 팀 편성 선언 + 작업 요약. 5~10줄 |
| **작업시간** | date range | 시작~종료 datetime. 소요시간 자동 계산용 |
| **소요시간** | formula | 작업시간 기반 자동. 편집 불요 |
| **결과** | select | 성공/부분성공/실패/판단보류 |
| **회고** | rich text | 3줄 고정: 잘된점 / 아쉬운점 / 다음 개선 |
| **양식버전** | select | v1.0/v2.0 — 신규는 반드시 v2.0 |
| **관련 기획/디자인/리서치** | relation | 해당 DB 엔트리 연결 (있으면) |

### 3-3. 작성 가이드

**이름**
- 좋음: "허브와이즈 편향지수 API 신규 구현"
- 나쁨: "API 작업", "수정함"

**내용**
팀 편성 선언 + 한 단락 요약:
```
[team] 작업: <작업명>
  패턴: β
  에이전트: planner → builder → tester
  스킬: requirements-spec, api-contract, qa-test

<작업 요약 3~5줄>
- 무엇을 했는가 (구체적으로)
- 어떤 결정을 했는가 (선택지와 근거)
- 무엇이 막혔는가 (있으면)
```

**결과 판정 기준**
- **성공**: 의도한 기능이 완전 작동, 테스트 통과
- **부분성공**: 기능 일부 작동, 혹은 일부 엣지 케이스 미해결. **학습 가치 최대**
- **실패**: 롤백 필요, 작동 안 함
- **판단보류**: 작업 완료됐지만 검증 전 (예: 샘플 관찰 필요)

**회고 3줄 (가장 중요)**
```
잘된점: <이번에 잘 된 것 1가지 구체적으로>
아쉬운점: <놓친 것, 오래 걸린 이유 1가지>
다음개선: <다음 유사 작업 때 할 것 1가지>
```

**부담스럽더라도 쓰자**. 이 3줄이 Phase 2에서 evolving-rules의 직접 입력이 된다.
"아무것도 아쉽지 않았음" 같은 무의미 텍스트는 쓰지 말 것 (공백이 낫다).

### 3-4. 소소한 세션 처리

10분 이내의 탐색·디버깅 세션은 기록 선택 (CTO 판단).
기록 생략 기준:
- 결과물이 없음 (단순 조회)
- 명확한 결론 없음
- 다른 세션의 준비 단계

## 4. 문서 메타데이터 표준

`docs/current/` 의 모든 문서는 상단에 다음 메타데이터 포함:

```markdown
# <문서 제목>

**양식**: <format-name> v<version>
**작성일**: YYYY-MM-DD
**수정일**: YYYY-MM-DD (수정 시마다)
**상태**: <draft|active|deprecated>
```

**상태 의미**:
- `draft`: 초안, 아직 시행 전
- `active`: 현재 적용 중
- `deprecated`: 더 이상 유효하지 않음 (asis로 이동 대기)

## 5. 파일 로그 양식 (v1.0)

`docs/logs/YYYY-MM/YYYY-MM-DD.md` 양식:

```markdown
# 2026-04-22 (화)

**양식**: daily-log v1.0

---

## 세션 1: <작업명>

**시간**: 13:00 ~ 15:30 (2h30m)
**프로젝트**: 하네스
**패턴**: δ
**결과**: 부분성공

### 내용
<작업 요약>

### 결정
<주요 결정>

### 회고
- 잘된점: ...
- 아쉬운점: ...
- 다음개선: ...

### 관련
- Notion devlog: <URL>
- 커밋: <hash>

---

## 세션 2: <작업명>
...
```

Notion에 쓰고 파일에도 append하는 건 **이중 작성 부담**이 있지만 다음 이유로 유지:
- 오프라인 접근
- git 히스토리로 시간 축 추적 가능
- Notion 장애 대비

Phase 2에서 Notion → 파일 자동 sync 스크립트로 이중 작성 부담 해소 예정.

## 6. 문서 라이프사이클

```
[신규 작성]
    ↓
[draft] → docs/current/ 에 초안 배치
    ↓ (시행 결정)
[active] → 상태 변경, 적용 중
    ↓ (버전업 or 폐기)
    ├─ 유형 A 버전업: asis/ 로 복사 후 current/ 덮어쓰기
    ├─ 유형 B 양식 진화: asis/ 로 복사 후 current/ 업데이트, 과거 기록 보존
    └─ 폐기: [deprecated] → asis/ 로 이동, current/ 에서 삭제
```

폐기된 문서는 asis에서 영구 보관. 삭제 금지.

## 7. Notion과 파일의 역할 분담

| 대상 | Notion (primary) | 파일 (보조) |
|---|---|---|
| devlog | ✅ 💻 개발 노트 | `docs/logs/` append |
| 키 관리 | ✅ 🔑 서비스/환경변수 | `.env.local` (값 pull, 커밋 금지) |
| 프로젝트 상태 | ✅ 📁 프로젝트 | — |
| 디자인 | Figma 파일 | — |
| 설계 결정·지침 | — | ✅ `docs/current/` |
| 에이전트 프롬프트 | — | ✅ `agents/` |
| 스킬 정의 | — | ✅ `skills/` |
| 훅 코드 | — | ✅ `hooks/` |
| 코드 | — | ✅ (git) |

**판정 기준**:
- 빈번 조회·필터·집계 필요 → Notion
- 버전 관리·git 추적 필요 → 파일
- 둘 다 필요 → 한쪽을 primary로 정하고 다른 쪽은 보조

## 8. 금지 사항

- **과거 기록 리트로핏 금지** (§2-2 참조)
- **`docs/current/` 외부에서 활성 문서 수정 금지** — 참조 경로 단일화
- **asis/ 수정 금지** (오타 포함)
- **로그와 지침을 같은 파일에 섞지 말 것**
- **Notion devlog 스키마를 임의 변경하지 말 것** — 양식 v3.0 승격은 CEO 합의 필요

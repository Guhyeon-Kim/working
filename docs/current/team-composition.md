# 팀 편성 지침 (Team Composition)

**양식**: team-composition v1.0
**작성일**: 2026-04-22

---

## 원칙

1. **기본값은 최소 팀.** 에이전트는 "증명된 만큼만" 호출한다. 기본 = Claude + Codex 2인.
2. **CTO(Claude)가 매 작업 시작 시 팀 구성을 선언한다.** 편성 근거는 학습 신호의 주 입력.
3. **룰이 아닌 판단 참조 패턴.** 아래 패턴은 "반드시 이렇게"가 아닌 "유사 작업 참고용".
4. **에이전트는 7개로 고정.** 성장은 스킬·훅·MCP의 추가·고도화로 일어난다.
5. **δ 패턴(하네스 자체 수정)은 에이전트 호출 금지.** 순환 참조 방지.

---

## 3층 아키텍처

### Layer 1: 공통 에이전트 7 (고정)

| ID | 역할 | 주 CLI | 보조 |
|---|---|---|---|
| `researcher` | 리서치·벤치마킹 | Gemini | insane-search, deep-research |
| `planner` | 요건정의·기획 고도화 | Claude | — |
| `copywriter` | 마케팅·카피 조정 | Gemini + Claude | — |
| `designer` | UI·정보구조 | Claude + Figma MCP | insane-design |
| `builder` | Front/Backend 구현 | Codex | Context7 MCP |
| `tester` | 테스트·디버깅 | Codex | Playwright MCP |
| `curator` | 컨텍스트·로그·메모리 | Claude | Notion MCP |

각 에이전트의 상세 프롬프트: `agents/<id>.md`

### Layer 2: 스킬·훅 카탈로그 (성장형)

- 스킬·훅은 에이전트에 묶이지 않는 **공용 자산**
- 어떤 에이전트든 필요 시 호출
- 성장은 카탈로그의 추가·고도화로 일어남

예: `builder`가 `error-handling` 스킬을 호출, `tester`가 `qa-test` 스킬을 호출.
다른 에이전트도 동일 스킬 호출 가능.

### Layer 3: 프로젝트 특화 (레포별 최소)

- 공통으로 해결 가능하면 Layer 1·2로 보냄
- 도메인 특화가 꼭 필요한 경우에만 프로젝트별 에이전트 신설
- 예시 후보 (현재 없음, 필요 시 신설):
  - `invest-domain` (허브와이즈 투자 도메인 특화)
  - `content-pipeline` (컨텐츠 자동화 파이프라인)

## 판단 축 3개

작업을 들어올 때 다음 3축으로 분류:

| 축 | 값 |
|---|---|
| **규모** | S (≤30분) / M (반나절) / L (하루+) / XL (주+) |
| **리스크** | 낮음 (UI·카피) / 중간 (로직·API) / 높음 (데이터·보안·결제) |
| **영역** | 제품 / 경량 / 컨텐츠 / 하네스본체 |

이 3축 값에 따라 전형 패턴(α~δ) 중 하나를 선택.

---

## 전형 패턴 4개

### α — 경량 단일 기능

**프로필**: S / 낮음 / 경량 (또는 제품의 아주 작은 수정)
**대표 작업**:
- B무거나에 새 생성기 1개 추가
- 허브와이즈 푸터 카피 수정
- 랜딩 페이지 색상 변경

**팀**: `builder` 단독
**스킵**: planner, researcher, designer, tester(스모크만), curator(기록은 세션 말미에 일괄)
**소요 예상**: 10~30분
**검증**: 배포 후 수동 스모크 1분 (클릭 테스트)

**편성 선언 예시**:
```
[team] 작업: 블레벨 "오늘의 간식 신탁" 생성기 추가
  패턴: α
  에이전트: builder
  스킬: error-handling
  스킵 사유: planner(요건 명확), researcher(도메인 익숙),
           designer(기존 페이지 양식 재사용), tester(스모크로 충분)
```

---

### β — 제품 신기능

**프로필**: M~L / 중간 / 제품 (사용자 데이터·API·상태 변화 관여)
**대표 작업**:
- 허브와이즈 편향지수 계산 API 신규
- 허브와이즈 공개 매매일지 필터 기능
- 사용자 인증·권한 확장

**팀**: `planner` → `builder` → `tester`
**조건부 추가**:
- `designer` (UI가 새로 필요한 경우)
- 개인정보·결제 관련 있으면 Claude 내부 보안 검토 체크리스트 수행 (별도 에이전트 없음, Claude 내장 판단)

**스킵**: researcher(사전조사 완료 상태 가정), copywriter(내부 기능이면 불요)

**소요 예상**: 2~8시간
**검증**: Playwright E2E (tester가 BLOCK/PASS 판정)

**편성 선언 예시**:
```
[team] 작업: 허브와이즈 편향지수 API 신규
  패턴: β
  에이전트: planner → builder → tester
  스킬: requirements-spec, api-contract, qa-test, error-handling
  스킵 사유: researcher(행동재무학 도메인 이미 정리됨),
           designer(UI 변경 없음), copywriter(내부 API)
```

---

### γ — 컨텐츠·자동화 튜닝

**프로필**: S / 낮음 / 컨텐츠 또는 자동화
**대표 작업**:
- 티스토리 포스트 생성 프롬프트 조정
- 인스타그램 해시태그 전략 업데이트
- 자동화 스크립트의 특정 규칙 변경

**팀**: `builder` 단독 + 샘플 출력 3개 수동 검증
**스킵**: 나머지 전부
**특이점**: `tester` 호출 안 함 (자동화 파이프라인은 E2E 어려움, 샘플 검증으로 대체)

**소요 예상**: 15분~1시간
**검증**: 샘플 출력 3개 중 2개 이상 품질 합격이면 통과

**편성 선언 예시**:
```
[team] 작업: 티스토리 "스틸러스 가이드" 생성 프롬프트 개선
  패턴: γ
  에이전트: builder
  스킬: library-docs
  스킵 사유: tester(샘플 3개 검증으로 대체), planner(규칙만 조정)
```

---

### δ — 하네스 자체 개선

**프로필**: M / 중간 / 하네스본체
**대표 작업**:
- 이 `CLAUDE.md` 수정
- `docs/current/` 문서 업데이트
- `agents/` 추가·수정
- `skills/` 신설

**팀**: **Claude 직접** (에이전트 호출 금지)
**조건부**: Codex를 stdlib 스크립트 구현 보조로만 사용 가능 (에이전트 형식 아님)

**이유**:
- 하네스 수정에 에이전트(builder 등)를 호출하면 그 에이전트가 참조해야 할
  하네스 자체가 변경 중 → 순환 참조 + 일관성 깨짐
- CTO가 직접 편집하는 것이 가장 안전·빠름

**소요 예상**: 30분~3시간
**검증**: Claude가 직접 읽고 정합성 확인 + 다음 세션에서 참조 여부 확인

**편성 선언 예시**:
```
[team] 작업: 하네스 v6.1 - 세션 시작 훅 추가
  패턴: δ
  에이전트: (호출 없음, Claude 직접)
  스킬: (해당 없음)
  스킵 사유: 순환 참조 방지
  주의: hooks/session-start.mjs 추가 후 settings.json 배선,
       검증 로그 필수
```

---

### custom — 패턴 외

어느 α~δ에도 맞지 않으면 `custom`으로 선언하고 자유 편성.
단 편성 근거를 상세히 기록 (다음 세션이 패턴으로 승격할지 판단).

**편성 선언 예시**:
```
[team] 작업: 허브와이즈 + 컨텐츠 자동화 교차 이벤트 (블로그 연동)
  패턴: custom
  에이전트: researcher → planner → builder (2개 레포 교차)
  스킬: api-contract, library-docs
  특이점: 두 프로젝트 동시 수정 필요. patch 전파 주의.
```

## 편성 선언 형식

매 작업 첫머리에 한 줄 선언. Notion devlog `내용` 필드에 그대로 포함.

```
[team] 작업: <작업명>
  패턴: <α|β|γ|δ|custom>
  에이전트: <호출 순서대로 나열>
  스킬: <호출 예정 스킬 나열>
  스킵 사유: <생략한 에이전트명 + 간단한 이유>
  [선택] 주의: <특이 사항>
```

**좋은 선언 예**:
```
[team] 작업: 허브와이즈 편향지수 API 신규
  패턴: β
  에이전트: planner → builder → tester
  스킬: requirements-spec, api-contract, qa-test
  스킵 사유: researcher(도메인 정리됨), designer(UI 변경 없음)
```

**나쁜 선언 예**:
```
[team] 작업: 뭐 좀 고침
  패턴: 대충
  에이전트: builder
```

→ 나중에 학습 신호로 쓸 수 없음. 선언은 **짧지만 구체적**이어야 함.

## Phase 진입 기준

Phase 1 (현재):
- `담당` 필드만 사용 (CEO/Claude/Gemini/Codex/수동/시스템)
- 편성 선언은 `내용` 텍스트 필드에 자유 기록

Phase 2 (검증 후):
- 별도 `team` 필드 (multi-select: 7 에이전트)
- `skills` 필드 (multi-select: 19개 스킬)
- `pattern` 필드 (select: α/β/γ/δ/custom)
- `links` 필드 (commit hash, PR URL 등)

Phase 2 진입 조건:
- v2.0 엔트리 ≥ 10건
- 패턴 α~δ 각 1회 이상 실사용
- CEO의 "편성 선언이 유용하다" 평가

## 자주 묻는 편성 케이스

### Q. builder와 tester 모두 Codex인데 분리하는 이유는?

**A.** 역할 분리가 프롬프트 품질을 높임. 같은 Codex라도:
- `builder`: "이 코드를 구현해줘" → 구현에 최적화
- `tester`: "이 코드의 엣지 케이스를 찾고 테스트해줘" → 검증에 최적화

하나의 세션에서 같은 코드를 "짰다가 테스트하다가" 하면 편향이 생김 (자기 코드에
관대). 분리 호출로 독립성 확보.

### Q. designer 필요 여부 판단 기준?

**A.** 다음 중 하나라도 해당되면 호출:
- 새 페이지/모달/플로우 신설
- 기존 UI의 정보 구조 변경
- 디자인 시스템 변경 (토큰, 간격, 색)
- Figma 파일 수정 동반

"기존 컴포넌트 재사용, 카피만 변경"은 호출 불요. builder가 직접 수정.

### Q. researcher와 planner 호출 순서?

**A.** 일반적으로 researcher → planner (정보 수집 → 요건 정리).
단 기존 정리된 도메인(허브와이즈의 행동재무학 등)이면 researcher 스킵.

### Q. copywriter는 언제 호출?

**A.** 외부 노출되는 카피(랜딩, 이메일, 푸시) 작성·개선 시. 내부 기능 명명
같은 건 불요.

### Q. curator 호출 시점?

**A.** 다음 상황:
- 세션이 길어져 컨텍스트 정리 필요
- 여러 세션에 걸친 프로젝트의 상태 요약
- Notion devlog 엔트리 상세 작성
- 메모리·로그 기반 "지난 2주 뭐 했지" 회고

일반 작업에서는 호출 불요. CTO(Claude)가 직접 처리.

## 편성의 학습 신호화

편성 선언은 학습 데이터로 축적된다:

- 같은 패턴(예: β)이 반복해서 성공 → 그 편성이 "최적"으로 강화
- 스킵한 에이전트가 나중에 필요했던 경우 → "스킵 금지" 패턴으로 역학습
- custom이 반복 등장 → 신규 패턴 ε 승격 검토

**장기적으로 이 데이터가 CTO의 판단 근거**가 된다. 편성 선언을 성의 있게 쓰는
것이 곧 하네스 지능화의 직접 원료.

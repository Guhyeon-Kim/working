# requirements-spec

**버전**: v6.0
**주 사용 에이전트**: planner
**연계 스킬**: planning, api-contract, wireframe-spec

---

## 목적

planner의 기획 메모를 builder·tester·designer가 작업할 수 있는 **요건서**로
정형화. 검증 가능한 수용 기준(Acceptance Criteria) 포함.

---

## 호출 시점

- β 패턴(제품 신기능) 작업 개시 전
- 기획 메모(planning 스킬 산출물) 승인 후
- 복잡한 기능 구현 전 스펙 고정 필요 시

---

## 입력

- 기획 메모 (planning 스킬 산출물)
- CEO 컨펌된 MoSCoW 분해

---

## 절차

### 1. 양식 채우기
요건서 템플릿(§출력)에 맞춰 정보 이식.

### 2. 수용 기준 작성
각 Must Have 항목에 Given/When/Then 또는 검증 가능한 기준:
- **Given**: 초기 상태 (사용자가 로그인된 상태에서)
- **When**: 사용자 행동 (계산하기 버튼 클릭)
- **Then**: 기대 결과 (5초 이내 FOMO 지수 0~100 범위 응답)

### 3. 엣지 케이스 추가
최소 5개:
- 데이터 없을 때
- 권한 없을 때
- 네트워크 느릴 때
- 입력 유효하지 않을 때
- 중복 요청

### 4. 비기능 요구사항
- 성능 (응답시간, 처리량)
- 보안 (인증·인가, 개인정보)
- 호환성 (브라우저·모바일)
- 접근성 (WCAG AA 기본)

### 5. CEO 컨펌 게이트
요건서 초안 → CEO에게 요약 제시 → 수정 반영 → 최종 승인

### 6. builder 전달 준비
- 요건서 파일 경로 명확
- 관련 디자인 링크 (designer 산출물)
- 관련 리서치 링크 (researcher 산출물)

---

## 체크리스트

- [ ] 모든 Must Have 항목에 수용 기준 있음
- [ ] 수용 기준이 측정·검증 가능함
- [ ] 엣지 케이스 최소 5개
- [ ] 비기능 요구사항 4가지 모두 채워짐
- [ ] 가정과 사실이 분리 명시됨
- [ ] Won't Have로 명시적 범위 제한됨
- [ ] CEO 컨펌 확인

---

## 금지

- **모호한 수용 기준**: "빠르게 동작해야 함" → "p95 응답시간 500ms 이내"
- **기술 구현 명시**: "Next.js Server Action으로 구현" → builder 영역 침범
- **UI 상세 디자인**: "버튼 색은 blue-600" → designer 영역 침범
- **범위 확장**: CEO가 말하지 않은 기능을 "이것도" 자체 추가

# Project Log 스킬

## 시니어 원칙

> project-log는 "한 일 목록"이 아니다. "다음 세션의 나 또는 다른 에이전트가 아무 질문 없이 이어받을 수 있게 하는 상태 기록"이다.
> 작업 완료 후 기록하면 이미 늦다. 시작 직후 기록해야 토큰 소진·세션 중단 시에도 상태가 보존된다.
> "등록된 작업 목록" ≠ 실행 큐. 보는 것과 실행하는 것은 다르다.

---

## 파일 위치

`.claude/project-log.md`

---

## 핵심 원칙

```
1. 작업 시작 즉시 [in-progress] 기록 (완료 후 기록 금지)
2. 단계 완료마다 "현재 단계" 업데이트 (어디서 멈춰도 복구 가능)
3. 세션 시작 시: 현황 파악용으로만 읽기
   → 작업 방향: CEO 지시 > 디자인 가이드 > project-log 등록 항목
   → 등록된 작업 목록을 보고 자동 실행하지 않음
4. 기술 부채는 발견 즉시 등록 (심각도 포함)
5. 롤백 발생 시 반드시 이력 기록 (원인 포함)
```

## 상태 코드

```
🔄 in-progress  → 현재 진행 중 (토큰 소진 시에도 보존)
⏸️ paused       → 컨펌 대기 또는 일시 중단 (재개 조건 명시)
✅ done         → self-review 포함 검증 완료
⏳ pending      → 선행 작업 미완료 또는 의존성 대기
🚨 urgent       → P0/P1 — 즉시 처리 (다른 작업 중단)
🔧 tech-debt    → 임시 처리, 추후 개선 필요 (심각도 필수)
🚫 blocked      → 외부 요인으로 진행 불가 (해제 조건 명시)
```

---

## 복잡도 기준

```
소: 파일 1~2개, 단순 수정, 텍스트·스타일 변경, 30분 이내
중: 파일 3~5개, 기능 수정, API 연동, 화면 추가, 2~4시간
대: 여러 시스템 영향, DB 스키마 변경, 신규 기능, 1일 이상
```

## 긴급 상황 처리 프로토콜

```
1. 현재 진행 중인 작업 상태 저장
   → project-log "현재 단계"에 정확히 기록 (어디까지 했는가)
   → 미완성 파일 있으면 명시

2. 현재 작업 ⏸️ paused 처리 (재개 조건 명시)

3. 긴급 작업 🚨 urgent로 등록 + 즉시 in-progress 전환

4. CEO에게 재조정 보고
   → "T-001 [단계]까지 완료 후 일시중단 — 긴급 T-005 착수"
```

## 세션 종료 시 필수 업데이트

```
□ 모든 in-progress 항목의 "현재 단계" 갱신
□ 이번 세션에서 발견한 기술 부채 등록
□ 이번 세션에서 내린 결정 → 결정 이력에 기록
□ done 항목 중 오래된 것 → project-log-old.md로 이동 (30일 기준)
□ 최종 업데이트 타임스탬프 갱신
```

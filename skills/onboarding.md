---

## 목적

새 프로젝트·레포·세션에 진입하는 사람(또는 AI)이 **최소 시간에 작업 가능한
상태**가 되도록 하는 온보딩 문서 작성. 독자는 "6개월 뒤의 자신"이 주 타깃.

---

## 호출 시점

- 새 프로젝트 레포 초기화 직후
- 기존 레포의 README·ONBOARDING이 낡았을 때
- 프로젝트 중대한 변경 후 (아키텍처·배포 방식 전환)
- 새 Claude 세션·협업자에게 "여기가 뭐하는 곳인지" 설명 필요 시

---

## 입력

- 대상 프로젝트·레포
- 독자 (새 협업자 / 미래의 자신 / 새 AI 세션)
- 기존 문서 (있으면)

---

## 절차

### 1. 독자 프로필 확정
- **기술 수준**: 이 분야 초심자인가, 중·상급인가
- **맥락 접근**: 프로젝트 히스토리 모름 / 일부 앎
- **목적**: 기여 / 사용 / 이해

대부분의 개인 프로젝트는 **6개월 뒤의 자신** 기준. 용어 설명·배경 포함.

### 2. 핵심 질문 답하기
독자가 처음 할 질문:
1. **이게 뭐?** (프로젝트가 무엇을 하는가)
2. **왜?** (존재 이유·해결하려는 문제)
3. **어떻게 실행?** (로컬 셋업·배포)
4. **어디서 뭐 찾아?** (디렉토리 구조)
5. **뭘 조심해야?** (함정·흔한 실수)
6. **막히면?** (도움 얻는 법)

### 3. 실행 가능성 우선
- 모든 명령어는 **실제 동작**해야 함 (placeholder 금지)
- 환경변수는 Notion 🔑 DB 링크와 함께
- 셋업 시간 예상 (5분 / 30분)

### 4. 구조화
- 상단에서 하단으로 "알아야 하는 순서"
- 불필요한 깊이 피하기 (한눈에 볼 수 있게)
- 핵심 링크는 최상단

### 5. 유지보수 가능성
- 자주 변하는 정보(버전·의존성)는 "확인 방법"만 기재
- 정적인 정보만 문서 본문에

## 7. 흔히 하는 실수 · Troubleshooting

### 로컬 서버 시작 안 됨
- `.env.local` 누락 → Notion 🔑 DB 재조회
- Node 버전 불일치 → `nvm use` (버전은 `.nvmrc` 참조)

### Supabase 연결 실패
- `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` 둘 다 필요
- RLS 정책으로 403 발생 시 → Supabase 대시보드에서 정책 확인

### 빌드 실패 (Vercel)
- TypeScript 오류: 로컬에서 `pnpm build` 먼저 확인
- 환경변수: Vercel 대시보드에 설정됐는지

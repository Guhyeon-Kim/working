# Wireframe 스킬

## 시니어 원칙

> 와이어프레임은 "화면을 그리는 것"이 아니다. "사용자가 목표를 달성하는 가장 짧은 경로"를 설계하는 것이다.
> 화면 하나에 사용자 목적은 하나여야 한다. 두 개가 보이면 설계가 잘못된 것이다.
> 와이어프레임 단계에서 발견되지 않은 UX 문제는 개발 후 10배 비싸게 수정된다.

## 작성 전 필수 확인

```
□ requirements.md 상태 = confirmed 인가?
  → draft 상태에서 와이어프레임 작성 시 전량 재작업 리스크
□ 기존 화면이 있다면 현재 구현 상태 파악 후 delta만 정의
□ design-spec.md(또는 design-system.md)의 레이아웃 제약 확인
  → content-area(1040px), 읽기 최대(720px), 광고 배치 규칙
```

## 화면 설계 원칙

### 정보 계층 설계 (F-패턴 / Z-패턴)
```
F-패턴 (콘텐츠 목록형 화면): 좌상단 → 우 → 좌하단 스캔
  적용: 대시보드, 종목 탐색, 뉴스 피드
  원칙: 첫 2줄에 핵심 정보 배치

Z-패턴 (랜딩/마케팅 화면): 좌상단 → 우상단 → 좌하단 → 우하단
  적용: 랜딩 페이지, 온보딩 화면
  원칙: 대각선 흐름에 CTA 배치
```

### 인지 부하 원칙
```
□ 화면당 의사결정 포인트 최대 3개 (Miller's Law)
□ Primary CTA는 화면당 1개 (두 개 이상 = 사용자 혼란)
□ 스크롤 없이 핵심 가치가 보여야 함 (Above the Fold)
□ 관련 정보는 시각적으로 그룹화 (Gestalt 원칙)
```

### 5-State 설계 (모든 화면 필수)
```
모든 화면은 다음 5가지 상태를 정의해야 한다:
1. 기본 상태 (데이터 있음, 로그인됨)
2. 로딩 상태 (Skeleton UI — 레이아웃 유지)
3. 빈 상태 (데이터 없음 — Empty State + CTA)
4. 에러 상태 (API 실패 — 재시도 버튼)
5. 부분 실패 상태 (일부 데이터만 로드)
```

```markdown
## 네비게이션 구조

```
[전체 화면 트리 — 부모/자식 관계와 경로 포함]
앱
├── 공개
│   ├── 랜딩 (/)
│   └── 로그인 (/login)
└── 인증 필요
    ├── 대시보드 (/dashboard)
    ├── 종목탐색 (/explore)
    │   └── 종목 상세 (/explore/[ticker])
    └── 설정 (/settings)
```

전환 규칙:
- 미인증 → 인증 필요 화면: /login?returnUrl=[경로]
- 인증 완료: returnUrl 있으면 복귀, 없으면 /dashboard

## 화면 정의

### [SC-001] 화면명 (/경로)

**목적**: [사용자가 이 화면에서 달성하는 단 하나의 목적]
**주요 사용자 행동**: [핵심 액션 1가지]
**성공 기준**: [이 화면이 잘 동작한다는 것을 어떻게 아는가]

**진입 경로**
- [어디서, 어떤 조건으로 이 화면에 오는가]
- 직접 URL 접근 가능 여부: O/X

**접근 권한**
- 공개 / 로그인 필요 / 프리미엄

**레이아웃 구조** (정보 계층 순서로 기술)
```
[헤더 영역]
  - 페이지 타이틀
  - 보조 액션 (필터, 정렬 등)

[콘텐츠 영역 — max-width: 1040px / 720px]
  - 핵심 정보 (Above the Fold)
  - 보조 정보
  - 광고 슬롯 (해당 시)

[하단 영역]
  - Primary CTA
  - 보조 링크
```

**5-State 정의**
| 상태 | 표시 내용 | UI 컴포넌트 |
|------|---------|-----------|
| 기본 | [데이터 있을 때] | [컴포넌트] |
| 로딩 | Skeleton 레이아웃 | Skeleton |
| 빈 상태 | "[안내 문구]" + [CTA] | EmptyState |
| 에러 | "데이터를 불러올 수 없습니다" + 재시도 | Alert + Button |
| 부분 실패 | [어느 부분이 실패해도 나머지는 표시] | Partial |

**액션 정의**
| 액션 | 트리거 | 결과 |
|------|-------|------|
| [버튼명] | 클릭 | [어디로 이동 / 무슨 일이 발생] |

**API 의존성**
- `GET /api/...` — 언제 호출, 무슨 데이터

**미결 사항**
- ❓ [결정 필요 사항]: [누가 결정]
```

## 와이어프레임 검증 체크리스트

```
□ 각 화면의 목적이 1개인가? (2개 이상 = 화면 분리 필요)
□ Primary CTA가 화면당 1개인가?
□ Empty State가 모든 데이터 목록 화면에 정의됐는가?
□ 모바일(375px) 기준으로도 핵심 정보가 스크롤 없이 보이는가?
□ 광고 슬롯이 콘텐츠 흐름을 방해하지 않는가? (08-ad-placement.md 기준)
□ 뒤로가기 시 어떤 상태로 돌아가는가? (Form dirty state 등)
□ 로딩 중 사용자가 다른 액션을 취하면?
□ 모든 화면이 flowchart.md의 플로우와 연결되는가?
```

## 작업 원칙

```
□ confirmed requirements.md 없으면 시작 금지
□ 디자인 요소(색상, 폰트 크기) 배제 — 레이아웃 구조만
□ 5-State 미정의 화면은 미완성으로 처리
□ 변경 시 이력 추가 + 관련 flowchart.md 영향 확인
□ design-agent에게 넘기기 전 체크리스트 전항목 통과 필수
```

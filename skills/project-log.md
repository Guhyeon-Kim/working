# project-log

**버전**: v6.0
**주 사용 에이전트**: curator
**연계 스킬**: context-summary, onboarding

---

## 목적

프로젝트의 **작업 로그·devlog·히스토리**를 구조화·유지보수. Notion 💻 개발
노트 DB를 primary로, 파일 로그를 보조로 운영.

---

## 호출 시점

- 세션 종료 시 devlog 엔트리 생성
- 주/월 단위 프로젝트 현황 리포트
- 과거 작업 재탐색 (디버깅 시 "전에 어떻게 해결했더라")
- 패턴 축적 (반복 이슈·성공 패턴)

---

## 입력

- 대상 프로젝트
- 작업 정보 (세션·기간)
- 소스 (채팅·커밋·PR·이슈)

## 절차

### 1. devlog 작성 (세션 종료 시)

세션 정보 수집:
- 작업 타이틀 (20자 내외)
- 시작·종료 시각
- 수행 에이전트·팀 편성
- 주요 결정·산출물
- 결과 판정
- 회고 3줄

Notion 엔트리 생성:
- 모든 필드 채움
- 관련 엔트리 relation 연결 (기획 DB 등)

파일 로그 append (해당 시):
- 당일 파일 없으면 생성 (`docs/logs/2026-04/2026-04-22.md`)
- 세션 섹션 append

### 2. 주간 리포트

매주 금요일 또는 CEO 요청 시:

**집계 대상**:
- 이번 주 devlog 엔트리 전체
- 프로젝트별 분리

**리포트 내용**:
- 프로젝트별 완료 작업
- 반복 이슈
- 성공률 (성공 / (성공+부분성공+실패))
- 다음 주 제안

### 3. 월간 리포트

매월 1일 또는 CEO 요청 시:

**추가 내용** (주간 + ):
- 월 전체 통계
- 패턴 후보 (반복 3회 이상 이슈)
- 회고 다음개선 항목 중 실행된 것 / 안 된 것
- 다음 월 초점 제안

### 4. 과거 재탐색

CEO 질문: "지난달 편향 지수 구현할 때 Supabase RLS 어떻게 했지?"

**절차**:
1. Notion 💻 개발 노트 검색 (키워드: 편향 지수, RLS)
2. 해당 엔트리 본문·회고 조회
3. 관련 PR·커밋 링크 따라가기
4. 요약 제공

## 출력 형식

```markdown
[team] 작업: 허브와이즈 편향 지수 API 신규
  패턴: β
  에이전트: planner → builder → tester

## 작업 요약
편향 4종(FOMO, 군중, 손실회피, 과신) 계산 엔드포인트 구현. Supabase RLS로
타인 데이터 접근 차단. ISR 1시간 캐시.

## 주요 결정
- calculateBiasScore 순수 함수 분리 (테스트 용이)
- Redis 대신 Next.js ISR 캐시 사용 (운영 복잡도 감소)
- Rate limit 미구현 (요건서에 수치 미확정, 후속 이슈로)

## 산출물
- PR: #123
- 커밋: abc1234, def5678
- 테스트: tests/e2e/bias-score.spec.ts

## 이슈
- 비활성 계정 500 에러 (tester 발견 → 수정)
```

## 금지

- **필드 누락**: 특히 `회고`, `결과`, `양식버전` 누락 시 학습 신호 잃음
- **모호한 회고**: "잘 됐다" 같은 내용 없는 회고
- **프로젝트 relation 생략**: 필터 불가해짐
- **과거 회고 수정**: 시점의 판단 보존 (리트로핏 금지)
- **임의 엔트리 삭제**: 상태 변경으로 lifecycle 관리

# Self Review 스킬

## 핵심 원칙

> 검증은 "잘 됐나 확인"이 아니라 "어떻게 잘못될 수 있는가"를 찾는 것이다.
> 작업자가 자신의 작업을 검증할 때 가장 흔한 실수는 기대한 것을 확인하는 것이다.
> 기대와 반대로 생각해라.

---

## 검증 적용 기준

| 복잡도 | 적용 | 기준 |
|--------|------|------|
| 소 | 1차 검증만 | 파일 1~2개, 단순 변경 |
| 중 | 1차 + 2차 | 파일 3~5개, 로직 변경 |
| 대 | 1차 + 2차 + 영향 범위 검토 | 여러 시스템 영향 |

---

## 1차 검증 체크리스트 (작업 에이전트 자체)

### 완성도 검증
```
□ 요청 내용이 모두 반영됐는가?
□ 빠진 항목이 없는가? (요구사항 번호 대조)
□ 명시적으로 제외 요청한 것이 구현에 포함됐는가? (범위 초과)
```

### 정확성 검증
```
□ 로직이 예상 입력에 대해 예상 출력을 내는가?
□ 엣지 케이스 (빈 값, 최댓값, 권한 없는 경우)는 처리됐는가?
□ 에러 상태에서 사용자가 복구할 수 있는가?
```

### 품질 검증
```
□ 공통 원칙 적용됐는가? (변경이력/타임스탬프 등)
□ 연관 파일 영향 있는가? → dependency-check 스킬 실행
□ 보안 취약점 없는가? (하드코딩 키, 권한 우회 등)
□ 성능 문제가 없는가? (N+1, 무한 루프, 메모이제이션 없는 무거운 계산)
```

### 규칙 준수
```
□ CLAUDE.md 핵심 규칙을 위반하지 않는가?
□ 금융 안전 정책 위반 표현이 없는가?
□ 사용자 확인이 필요한 결정을 임의로 처리하지 않았는가?
```

## 2차 검증 체크리스트 (context-agent 교차)

```
□ 전체 프로젝트 맥락과 일치하는가?
□ 다른 에이전트 산출물과 충돌 없는가?
□ 누락된 연관 작업 없는가?
□ 다음 에이전트가 작업하기 충분한 정보인가?
□ 과거 반복 실수 패턴 (failure-cases.md)에 해당하지 않는가?
```

---

## "반대로 생각하기" 검증 (대형 작업)

```
□ 이 작업이 기존에 동작하던 것을 망가뜨릴 수 있는가?
□ 다른 사용자나 역할에서 이 기능이 어떻게 보이는가?
□ 네트워크 오류/서버 다운 상황에서 어떻게 동작하는가?
□ 데이터가 없는 상태에서 어떻게 동작하는가?
□ 악의적인 사용자가 이 기능을 어떻게 남용할 수 있는가?
```


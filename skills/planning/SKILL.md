# Planning 스킬

## 핵심 원칙

> 요구사항은 "무엇을 만들 것인가"가 아니라 "무엇을 만들지 않을 것인가"도 포함한다.
> 측정할 수 없는 요구사항은 요구사항이 아니다.
> 모든 요구사항에는 비즈니스 가치와 구현 비용이 있다. 이 둘을 비교해야 우선순위가 나온다.

## 우선순위 결정 매트릭스

```
사용자 가치 높음 + 구현 비용 낮음 → P1 필수 (즉시)
사용자 가치 높음 + 구현 비용 높음 → P2 핵심 (이번 구현)
사용자 가치 낮음 + 구현 비용 낮음 → P3 선택 (다음 스프린트)
사용자 가치 낮음 + 구현 비용 높음 → 명시적 제외
```

## 작업 원칙

```
□ 변경 시 새 파일 생성 금지 → 이력 추가
□ "현재 적용 상태" 항상 최신 유지
□ 모든 NFR은 수치로 정의 (막연한 "빠르게" 금지)
□ 명시적 제외 항목 필수 — 안 하는 것도 결정이다
□ confirmed 전에 design/dev 시작 금지
□ 작업 범위 외 발견 시 project-log pending 등록
```

## 좋은 요구사항의 3가지 기준

1. **명확성**: 두 사람이 읽었을 때 같은 의미로 이해해야 함
2. **완전성**: 정상 케이스 + 에러 케이스 + 엣지 케이스 모두 포함
3. **검증 가능성**: 완료 여부를 객관적으로 판단할 수 있어야 함

## 나쁜 요구사항 패턴 (피해야 할 것)

```
❌ "빠르게 로드되어야 한다" → 기준 없음
✅ "초기 로드 3초 이내, LCP 2.5초 이내"

❌ "사용자 친화적인 에러 처리" → 모호함
✅ "API 실패 시 '잠시 후 다시 시도해주세요' 메시지 + 재시도 버튼 표시"

❌ "필요하면 로그인 처리" → 조건 불명확
✅ "비로그인 사용자가 접근 시 /login으로 리다이렉트"
```

## 엣지 케이스 체크리스트

```
□ 데이터가 없을 때 (빈 목록, 0원, null)
□ 데이터가 너무 많을 때 (긴 종목명, 많은 보유 종목)
□ API/네트워크 실패 시
□ 권한 없는 사용자 접근 시
□ 이미 처리된 작업 중복 요청 시
□ 모바일 환경 (터치, 작은 화면)
```

### 금융 서비스 필수 고지

추천/전략 기능은 반드시 포함:
- WHY: 왜 이 전략인가

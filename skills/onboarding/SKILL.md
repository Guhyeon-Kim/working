# Onboarding 스킬

## 시니어 원칙

> 온보딩은 "파일을 만드는 것"이 아니다. "프로젝트의 현재 상태를 정확히 진단하고, 다음 세션이 아무 마찰 없이 시작될 수 있게 만드는 것"이다.
> 기존 파일을 덮어쓰는 순간 이전 컨텍스트가 사라진다. 항상 보완한다.
> 파악되지 않은 코드베이스를 건드리는 것은 지뢰밭을 걷는 것이다. 진단 먼저, 행동 나중.

---

## 실행 조건

- `project-log.md` 없을 때 (신규 프로젝트)
- `CLAUDE.md` 없을 때
- 신규 세션에서 "이어서 해줘", "현재 상태 알려줘" 요청 시
- 기존 프로젝트인데 컨텍스트가 불명확할 때

---

## 실행 순서

### Step 1: 현재 상태 진단 (읽기 전용 — 변경 금지)

```bash
# 기술 스택 파악
ls -la                          # 루트 구조
cat package.json 2>/dev/null    # 프론트엔드 스택
cat requirements.txt 2>/dev/null  # 백엔드 스택 (Python)
cat pyproject.toml 2>/dev/null  # 백엔드 스택 (Poetry)

# 환경 파악
ls .env* 2>/dev/null            # 환경변수 파일 목록 (내용은 읽지 않음)
ls -la .claude/ 2>/dev/null     # 기존 문서 여부

# Git 상태 파악
git log --oneline -5            # 최근 작업 맥락
git status                      # 현재 변경 사항
git branch                      # 현재 브랜치
```

진단 후 다음을 확인한다:
```
□ 기술 스택: [프론트엔드 / 백엔드 / DB / 배포]
□ 기존 CLAUDE.md: 있음/없음
□ 기존 project-log.md: 있음/없음
□ 기존 .claude/docs/: 있음/없음 (있으면 내용 파악)
□ .claude/docs/design-system.md: 있음/없음 (없으면 design-system 스킬로 생성 안내)
□ 마지막 커밋: [날짜 + 메시지]
□ 미완성 작업: git status 기반
□ 환경변수 파일: [있음/없음 — 내용 열람 금지]
```

### Step 2: 리스크 평가

```
기존 코드베이스가 있는 경우:
□ 의존성 오래됐는가? (package.json의 last modified 확인)
□ 환경변수 파일이 .gitignore에 있는가?
□ 테스트 코드가 있는가? (있으면 실행 환경 파악)
□ 기술 부채 표시 (TODO/FIXME/HACK 주석) 있는가?

신규 프로젝트인 경우:
□ 기술 스택 선택 근거가 있는가?
□ 외부 서비스 의존성 확인 (API 키 필요 여부)
□ 배포 환경 결정됐는가?
```

### Step 3: 문서 생성/보완

**기존 파일이 있으면 내용 보완만 — 덮어쓰기 금지**

#### CLAUDE.md 생성 (없는 경우만)


## 기존 프로젝트 진단 보고서 형식

신규 프로젝트가 아닌 기존 코드베이스 온보딩 시:

```
🔍 코드베이스 진단 완료 [YYYY-MM-DD HH:MM]

## 기술 스택
- Frontend: [스택 + 버전]
- Backend: [스택 + 버전]
- DB: [종류]
- 배포: [플랫폼]

## 현재 상태
- 마지막 작업: [날짜] — [커밋 메시지]
- 미완성 변경: [있음 N개 / 없음]
- 진행 중이던 작업: [있음 → 내용 / 없음]

## 발견된 리스크
- [리스크 1]: [심각도] — [권고 조치]
- [리스크 2]: ...

## 기술 부채 (코드에서 발견)
- [TODO/FIXME 내용]: [파일:라인]

## 다음 권고 작업
1. [가장 먼저 해야 할 것]
2. [그 다음]

무엇부터 시작할까요?
```

## 주의사항

```
□ 기존 파일 덮어쓰기 절대 금지 → 내용 보완만
□ .env 파일 내용 열람 금지 (목록만 확인)
□ 의존성 설치, 빌드, 테스트 실행 금지 (사용자 승인 후)
□ "아마 이럴 것 같다" 추측 금지 → 코드로 확인 후 보고
□ 기존 project-log에 등록된 작업 자동 실행 금지
   → 현황 파악용으로만 읽고, 작업 방향은 CEO 지시 우선
```

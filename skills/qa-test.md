# qa-test

**버전**: v6.0
**주 사용 에이전트**: tester
**연계 스킬**: self-review, review-request

---

## 목적

테스트 케이스 설계의 체계적 체크리스트. 누락 없이 happy path·엣지 케이스·
에러 케이스·보안·성능을 커버.

---

## 호출 시점

- tester가 테스트 설계 시작할 때
- 기존 테스트 케이스 보강 시 (리그레션 방지)

---

## 입력

- 요건서 (수용 기준 포함)
- builder의 구현 완료 보고
- 변경 파일 목록

---

## 절차

### 1. 수용 기준 → 테스트 케이스 매핑
각 Must Have 수용 기준에 1:N 테스트 케이스 대응.

### 2. 시나리오 카테고리별 설계

#### Happy Path (1~2개)
- 정상적인 주 사용 흐름

#### 엣지 케이스 (3~5개)
- 빈 입력 (null, undefined, "", [])
- 최대 크기 (매우 긴 문자열, 큰 배열)
- 경계값 (0, -1, MAX_INT, MIN_INT)
- 특수 문자 (한글, 이모지, SQL 특수문자)
- 동시성 (동일 요청 여러 건)
- 타이밍 (네트워크 지연, 빠른 연타)

#### 에러 케이스 (2~3개)
- 잘못된 입력 (잘못된 형식·범위)
- 권한 없음 (미인증·부족한 role)
- 외부 의존성 실패 (DB·API down)
- Timeout

#### 보안
- SQL Injection 시도
- XSS 시도
- CSRF (해당 시)
- 타인 데이터 접근 시도
- Rate limit (해당 시)

#### 성능 (수용 기준 있을 시)
- 응답 시간 p50·p95·p99
- 대용량 데이터 (1000·10000 레코드)
- 동시 요청 (10·100 동시)

#### UI (해당 시)
- 로딩 상태 표시
- 빈 상태 표시
- 에러 표시
- 반응형 (모바일·데스크톱)
- 키보드 접근

### 3. 테스트 타입 분류
- **단위 테스트**: 순수 함수·로직
- **통합 테스트**: API 엔드포인트 전체
- **E2E 테스트**: 사용자 관점 플로우

### 4. 테스트 케이스 문서화
실제 코드 작성 전 케이스 목록 확정.

### 5. 실행 환경 준비
- 시드 데이터
- 테스트 계정
- 모킹 전략

```markdown
## 1. 수용 기준 매핑

| # | 수용 기준 | 테스트 케이스 # |
|---|---|---|
| AC-1 | 5초 이내 응답 | TC-001, TC-010 |
| AC-2 | 0~100 범위 | TC-002, TC-011 |
| AC-3 | 인증 필수 | TC-020, TC-021 |

## 2. 테스트 케이스

### Happy Path
- TC-001: 정상 로그인 사용자가 편향 지수 조회 성공
- TC-002: 계산된 지수가 0~100 범위 내

### 엣지 케이스
- TC-010: 매매 기록 0개 → 204 No Content
- TC-011: 매매 기록 1000개 → 응답 시간 내
- TC-012: 빈 body 요청 → 400 Bad Request
- TC-013: 이모지 포함 종목명 → 정상 처리
- TC-014: 연속 5회 빠른 호출 → 중복 계산 없이 동일 응답

### 에러 케이스
- TC-020: 미인증 요청 → 401
- TC-021: 만료된 JWT → 401
- TC-022: 비활성 계정 → 403
- TC-023: DB 연결 실패 → 503 (시뮬레이션)

### 보안
- TC-030: SQL Injection 시도 (`period=1m' OR 1=1--`) → 400
- TC-031: 타 사용자 데이터 접근 시도 → 403 (RLS)
- TC-032: Rate limit 11번째 호출 → 429

### 성능
- TC-040: p95 응답 시간 < 3초 (N=100)
- TC-041: 동시 10 요청에서 모두 성공

## 3. 테스트 타입
- 단위: TC-002 (calculateBiasScore 함수)
- 통합: TC-001, TC-010~023
- E2E: TC-001 (사용자 관점 전체 흐름)

## 4. 환경
- 테스트 DB: Supabase dev
- 시드: scripts/seed-bias-test.ts
- 테스트 계정:
  - active@test.com (정상)
  - inactive@test.com (비활성)
  - premium@test.com (rate limit 높음)
```

## 금지

- **Happy path만 테스트**: 프로덕션에서 엣지 케이스 폭발
- **False positive 수용**: assertion 없이 "실행만 성공"하면 통과 처리 금지
- **임의 누락**: 시간 없다고 케이스 줄이기 (리포트에 명시해야 함)
- **프로덕션 데이터 사용**: dev·staging만

# Requirements Spec

개발팀이 추가 질문 없이 구현할 수 있는 요구사항을 작성하는 방법.

## 좋은 요구사항의 3가지 기준

1. **명확성**: 두 사람이 읽었을 때 같은 의미로 이해해야 함
2. **완전성**: 정상 케이스 + 에러 케이스 + 엣지 케이스 모두 포함
3. **검증 가능성**: 완료 여부를 객관적으로 판단할 수 있어야 함

## 나쁜 요구사항 패턴 (피해야 할 것)

```
❌ "빠르게 로드되어야 한다" → 기준 없음
✅ "초기 로드 3초 이내, LCP 2.5초 이내"

❌ "사용자 친화적인 에러 처리" → 모호함
✅ "API 실패 시 '잠시 후 다시 시도해주세요' 메시지 + 재시도 버튼 표시"

❌ "필요하면 로그인 처리" → 조건 불명확
✅ "비로그인 사용자가 접근 시 /login으로 리다이렉트"
```

## 엣지 케이스 체크리스트

요구사항 작성 후 아래를 반드시 확인:

```
□ 데이터가 없을 때 (빈 목록, 0원, null)
□ 데이터가 너무 많을 때 (긴 종목명, 많은 보유 종목)
□ API/네트워크 실패 시
□ 권한 없는 사용자 접근 시
□ 이미 처리된 작업 중복 요청 시
□ 모바일 환경 (터치, 작은 화면)
```

## HubWise 공통 패턴

### 권한 매트릭스 기본값

| 기능 유형 | 게스트 | 회원 | Admin |
|----------|--------|------|-------|
| 공개 콘텐츠 조회 | O | O | O |
| 개인 데이터 조회 | X (로그인 유도) | O (본인만) | O (전체) |
| 데이터 생성/수정 | X | O | O |
| Admin 기능 | X | X | O |

### 금융 서비스 필수 고지

추천/전략 기능은 반드시 포함:
- WHY: 왜 이 전략인가
- RISK: 어떤 리스크가 있는가
- CONDITION/EXIT: 언제 진입/청산하는가

# Context Summary 스킬

## 시니어 원칙

> 컨텍스트 파악은 "파일을 읽는 것"이 아니다. "지금 시스템이 어떤 상태인가"를 진단하는 것이다.
> 읽은 것을 그대로 요약하면 사서다. 읽고 판단하면 CTO다.
> CLAUDE.md는 항상 최신 상태만 유지한다. 이력은 project-log의 역할이다.

---

## 읽기 순서 (항상 이 순서 준수)

```
1. project-log.md → 어디까지 했는가, 무엇이 블록됐는가
2. CLAUDE.md → 프로젝트 기본 컨텍스트 + 핵심 규칙
3. docs/planning/05-design-system.md 존재 확인
   → 존재하면: 모든 UI 작업은 이 문서 기준
   → project-log의 "등록된 작업" 목록은 참조만 (자동 실행 금지)
4. 작업 관련 산출물 파일 선택적 로드 (아래 기준)
```

---

## 작업 방향 결정 우선순위 (절대 준수)

```
CEO 지시 > docs/planning/05-design-system.md > project-log 등록 항목

⚠️ project-log에 등록된 작업을 보고 자동 실행하는 행위 금지
⚠️ 소항목 패치가 아닌 "페이지 단위 전면 개편"이 최소 작업 단위
⚠️ design-system.md가 존재하면 소항목들은 이미 구식일 수 있음
```

---

## 선택적 파일 로드 기준

```
요구사항 관련 작업      → docs/planning/01-product-specification.md
화면/UI 관련 작업       → docs/planning/05-design-system.md + wireframe.md
API/데이터 관련 작업    → docs/planning/04-database-schema.md + api-spec.md
보안 관련 작업          → .claude/docs/security-report-latest.md
배포 관련 작업          → .claude/docs/staging-setup.md
콘텐츠 관련 작업        → project-log.md (dev-posts 상태 확인)
전체 파악 필요          → 전부 로드 (세션 시작 첫 번째에만)
```

---

## 시스템 상태 진단 (세션 시작 시)

단순히 파일을 요약하는 것이 아니라 다음을 판단한다:

```
□ 전 세션에서 미완성으로 끊긴 작업이 있는가?
  → in-progress 항목 + "현재 단계" 확인
  → 재개 가능한지, 먼저 확인이 필요한지 판단

□ 블록된 작업의 해제 조건이 충족됐는가?
  → blocked 항목 확인 + 현재 시점에서 가능한지 판단

□ 기술 부채 중 HIGH 심각도 항목이 있는가?
  → 새 기능 작업 전에 먼저 처리 권고

□ 마지막 배포 이후 검증되지 않은 변경이 있는가?
  → git log와 qa-results 비교

□ 환경 상태가 정상인가?
  → Gemini CLI, Codex CLI 상태 확인 필요 여부
```

## CLAUDE.md 업데이트 원칙

```
추가 X → 덮어쓰기 O (항상 최신 상태만)
상세 이력 → project-log.md가 담당
결정 기록 → project-log.md "결정 이력" 섹션
CLAUDE.md → 요약본 + 규칙만 (200줄 이하)

업데이트 트리거:
- 새로운 핵심 규칙 추가 시
- 기술 스택 변경 시
- 반복 실수 패턴 발견 시
- 문서 인덱스 변경 시
```

## 주의사항

```
□ project-log "등록된 작업" 목록을 보고 자동 실행 금지
□ CLAUDE.md 200줄 초과 금지 — 초과 시 상세 내용 삭제 or 다른 파일로 이동
□ "아마 이렇게 됐을 것 같다" 추측 금지 → 코드/git으로 확인
□ 전체 파일 파악이 필요 없는 경우 선택적 로드 (토큰 절약)
□ CLAUDE.md 업데이트 시 "추가"가 아닌 "최신 상태로 교체"
```

# Review Request 스킬

## 시니어 원칙

> 컨펌 요청은 "내가 한 일을 보고하는 것"이 아니다. 사용자가 "YES"를 누르면 무슨 일이 일어나는지를 명확히 알 수 있게 하는 것이다.
> 리스크가 높은 결정일수록 더 간결해야 한다. 길면 읽지 않는다.
> 모르면 물어봐라. 추측한 채로 완료 처리하는 것이 가장 비싼 실수다.

---

## 컨펌 필요 여부 판단 기준

컨펌 없이 자체 처리할 수 있는 것과 반드시 확인해야 하는 것을 구분한다.

| 구분 | 자체 처리 가능 | 반드시 컨펌 |
|------|-------------|-----------|
| 되돌릴 수 있는가? | O → 자체 처리 | X → 컨펌 |
| 다른 사람에게 영향? | 없음 → 자체 처리 | 있음 → 컨펌 |
| 비용/보안 영향? | 없음 → 자체 처리 | 있음 → 컨펌 |
| 사용자 확인 필요한 판단? | 명확 → 자체 처리 | 모호 → 컨펌 |

## 보고 대상

| 보고 대상 | 방법 |
|----------|------|
| CEO (사용자) | 채팅 직접 보고 |

## 에이전트별 컨펌 기준 (위계 포함)

```
반드시 컨펌 (CEO 판단 필요)
├── planning-agent  → 요구사항 확정, 화면 목록, 명시적 제외 항목
├── design-agent    → UI 레이아웃 방향, 디자인 시스템 변경
├── research-agent  → 리서치 결과 + 권고 방향 (최종 결정은 CEO)
├── security-agent  → Critical/High 취약점 발견 시 즉시 에스컬레이션
└── qa-agent        → 배포 최종 승인 (staging 검증 완료 후)

완료 보고만 (자체 판단 항목)
├── data-agent      → DB 설계, API 명세 (요구사항 기반 확정 작업)
├── frontend-agent  → 코드 구현, 버그 수정 (설계 확정된 범위 내)
├── backend-agent   → API 구현 (api-spec.md 기준)
├── context-agent   → project-log 갱신, 브리핑
└── infra-agent     → 성능 분석, 비용 분석 보고

예외: 아래 상황은 에이전트 유형 관계없이 항상 컨펌
- DB 스키마 삭제/변경 (zero-downtime 아닌 경우)
- 배포 대상이 production인 경우
- 비용 증가 예상 ($10/월 이상)
- 보안 게이트 우회가 필요한 경우
```

## 리스크 수준별 보고 형식

### Level 1 — 낮은 리스크 (단순 완료 보고)
```
[에이전트] [작업명] 완료 [YYYY-MM-DD HH:MM]
- [1줄 요약]
확인 필요하시면 말씀해주세요.
```

### Level 2 — 중간 리스크 (컨펌 요청)
```
[에이전트] [작업명] 검토 요청 [YYYY-MM-DD HH:MM]

변경 내용
- [구체적 변경 1]
- [구체적 변경 2]
- [구체적 변경 3] (최대 3줄)

영향 범위: [어느 화면/기능/사용자에게 영향]

진행할까요? (YES / NO / 수정 필요)
```

### Level 3 — 높은 리스크 (리스크 명시 컨펌)
```
⚠️ [에이전트] [작업명] — 리스크 컨펌 필요 [YYYY-MM-DD HH:MM]

변경 내용
- [변경 내용]

리스크
- [무엇이 달라지는가]
- [되돌리기 어려운 이유 / 영향 범위]

대안 (있는 경우)
- A안: [설명] — 리스크: [낮음/높음]
- B안: [설명] — 리스크: [낮음/높음]

권고: [A안 / B안 / 재검토] — 이유: [한 줄]

승인하시겠습니까? (YES / NO / 대안 검토)
```

### Level 4 — 긴급 에스컬레이션 (즉시 중단)
```
🚨 긴급 — [에이전트] 작업 중단 [YYYY-MM-DD HH:MM]

발견 내용: [무엇을 발견했는가]
중단 이유: [왜 계속 진행할 수 없는가]
현재 상태: [코드/DB/배포 상태 — 롤백 필요 여부]

즉시 확인 필요합니다.
```

## 보고 품질 체크리스트 (전송 전)

```
□ 기술 용어를 사용했다면 괄호로 설명했는가?
□ "YES를 누르면 무슨 일이 일어나는가"가 명확한가?
□ 되돌릴 수 없는 작업이라면 명시했는가?
□ 3줄 초과 설명이 있다면 더 압축할 수 없는가?
□ "확인해주세요" 같은 모호한 표현 대신 YES/NO/수정필요로 끝나는가?
□ 미결 사항을 "됐을 것 같다"로 추측 처리하지 않았는가?
```

---

## 금지 패턴

```
❌ "혹시 괜찮으시면 확인 부탁드립니다" → 컨펌이 필요한지 아닌지 명확히 할 것
❌ "최선을 다했습니다" → 결과만 말할 것
❌ 3개 이상의 질문을 한 번에 → 가장 중요한 1개만
❌ 전체 코드/문서 재출력 → 변경된 부분만
❌ "문제없이 완료됐습니다" (검증 안 된 채로) → self-review 통과 후 보고
```

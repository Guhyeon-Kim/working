# Flowchart 스킬

## 시니어 원칙

> 플로우차트는 "잘 되는 경우"를 그리는 게 아니다. "어디서 사용자가 이탈하는가", "어떤 에러가 어디서 발생하는가"를 설계하는 것이다.
> 정상 흐름은 전체의 30%다. 나머지 70%는 분기, 오류, 재시도, 취소, 권한 실패다.
> 플로우 하나로 요구사항의 진실성이 드러난다. 그릴 수 없는 플로우는 확정되지 않은 요구사항이다.

```
□ requirements.md "현재 적용 상태" = confirmed 인가?
  → draft 상태에서 플로우 작성 시 전량 폐기 리스크
□ wireframe.md의 화면 목록과 플로우의 화면이 일치하는가?
□ 이미 구현된 기능이라면 실제 코드의 분기 조건과 일치하는가?
```

## 플로우 설계 원칙

### 분기 조건 명시 기준
모든 분기는 **사용자 상태 + 시스템 상태 + 데이터 상태** 3축으로 정의한다.

```
나쁜 예: [로그인 여부 확인] → YES/NO
좋은 예:
  조건: Supabase 세션 유효 + access_token 만료 여부
  YES (세션 유효): → 대시보드
  NO (세션 없음): → 로그인 페이지
  NO (토큰 만료): → refresh 시도 → 실패 시 로그인 페이지
```

### 필수 포함 시나리오 (매 플로우)
```
1. Happy Path (정상 흐름) — 가장 짧아야 함
2. Sad Path (사용자 실수 — 재시도 가능)
3. Error Path (시스템 오류 — 재시도 or 지원 요청)
4. 권한 없음 경로
5. 세션 만료 경로 (인증 필요 기능만)
6. 데이터 없음 경로 (Empty State)
```

```markdown
## [FL-001] 플로우명

### 메타
- 시작 진입점: [어디서 시작하는가]
- 종료 상태: [성공 종료 / 실패 종료]
- 관련 API: [엔드포인트]
- 관련 DB 테이블: [테이블명]
- 최대 허용 소요 시간: [N초 — NFR 기준]

### Happy Path
```
[진입 조건: 로그인 상태, 권한 O]
  ↓
[화면: 입력 폼 — SC-001]
  ↓ [사용자 액션: 제출]
[프론트 유효성 검사]
  ↓ 통과
[API 호출: POST /resource]
  ↓ 200 OK
[성공 피드백: 토스트 메시지]
  ↓
[화면 이동: 목록 — SC-002]
```

### Sad Path (사용자 오류)
```
[화면: 입력 폼 — SC-001]
  ↓ [제출]
[프론트 유효성 검사]
  └─ 실패: 필수 입력값 누락
      → 인라인 에러 표시 + 포커스 이동
      → 재입력 유도 (재시도 가능)
```

### Error Path (시스템 오류)
```
[API 호출: POST /resource]
  ├─ 400: 입력 데이터 형식 오류 → 폼 에러 표시
  ├─ 401: 세션 만료 → refresh 시도 → 로그인 리다이렉트
  ├─ 403: 권한 없음 → "접근 권한이 없습니다" + 이전 화면
  ├─ 429: Rate limit → "잠시 후 다시 시도" + N초 타이머
  └─ 500: 서버 오류 → 재시도 버튼 + 지원 문의 링크
```

### 권한 없음 경로
```
[미인증 사용자 직접 접근]
  → 로그인 페이지 리다이렉트 (returnUrl 파라미터 포함)
[인증O but 권한X (일반→프리미엄 전용)]
  → "프리미엄 기능입니다" 업셀 모달
```

### 데이터 없음 경로
```
[화면 로드 — 데이터 없음]
  → Empty State UI (아이콘 + 안내 문구 + CTA 버튼)
  → [첫 데이터 생성 플로우로 연결]
```

### 예외 처리 매트릭스
| 상황 | 원인 | 사용자 메시지 | 시스템 액션 |
|------|------|-------------|------------|
| [상황] | [기술 원인] | [사용자 노출 문구] | [로그/알림/재시도] |

### 미결 사항
- [❓ 확인 필요 사항]: [누가 결정해야 하는가]
```

## 복잡도 판단 기준

```
낮음: 분기 3개 이하, 외부 API 없음, 화면 2개 이하
중간: 분기 4~7개, API 호출 1~2개, 화면 3~5개
높음: 분기 8개 이상, API 3개 이상, 외부 서비스 연동, 권한 분기 복수
```

복잡도 **높음** 플로우는 반드시 CEO 확인 후 confirmed 처리한다.

## 플로우 검증 체크리스트

```
□ 모든 분기의 두 방향이 모두 정의됐는가? (한쪽만 그린 경우 = 미완성)
□ 오류 경로에서 사용자가 스스로 복구 가능한가?
□ "뒤로가기" 버튼이 어떤 상태를 만드는가? (부분 저장 문제)
□ 네트워크 타임아웃(30초) 시 어떻게 동작하는가?
□ 동일 작업을 두 번 실행 시 (이중 클릭) 어떻게 방지하는가?
□ 모바일에서 백그라운드 → 복귀 시 세션 상태는?
□ 감사 로그가 필요한 분기점에 logging 이벤트가 명시됐는가?
```

## 작업 원칙

```
□ requirements.md confirmed 후에만 작성 시작
□ 정상 경로 1개 = 미완성 — 오류/예외 포함해야 완성
□ 분기 조건에 "사용자 행동"뿐 아니라 "시스템 상태"도 명시
□ 변경 시 이력 추가 + 관련 wireframe.md 영향 여부 확인
□ "그릴 수 없는 흐름"은 요구사항이 불확실한 신호 → 즉시 escalate
```

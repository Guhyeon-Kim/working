# design-review

**버전**: v6.0
**주 사용 에이전트**: designer
**연계 스킬**: design-system, ui-component

---

## 목적

기존 UI·Figma 파일·랜딩 페이지를 체계적으로 리뷰하여 개선점을 도출. 감에
기대지 않는 체크리스트 기반.

---

## 호출 시점

- 프로젝트의 기존 UI 개선 요청 시
- 디자인 시스템 정비 전 현황 파악
- 경쟁사·레퍼런스 UI 분석
- 자체 디자인 제안 자가 검증

---

## 입력

- 리뷰 대상: Figma URL 또는 배포된 페이지 URL
- 리뷰 범위: 전체 / 특정 섹션 / 특정 플로우
- 비교 기준: 디자인 시스템 준수 / 사용성 / 접근성 / 일관성

---

## 절차

### 1. 맥락 파악 (5분)
- 이 화면의 목적
- 대상 사용자
- 이전 화면·다음 화면과의 관계

### 2. 체크리스트 기반 리뷰
아래 관점 각각 체크.

### 3. 우선순위 매기기
이슈를 심각도로 분류:
- **Critical**: 사용 불가능하게 하는 문제
- **High**: 사용성 크게 저해
- **Medium**: 개선 여지
- **Low**: 취향·선호 수준

### 4. 개선 제안
각 이슈에 구체적 개선안 (말로만 "개선 필요" 금지).

---

## 리뷰 관점

### 정보 위계
- [ ] 가장 중요한 정보가 시각적으로 돋보이는가
- [ ] 3단계 이내의 위계인가 (과도한 중첩 X)
- [ ] 스크롤 없이 핵심 메시지 도달 가능한가

### 일관성
- [ ] 색 토큰 준수 (임의 hex 값 X)
- [ ] 간격 토큰 준수 (4의 배수 등)
- [ ] 타이포그래피 일관 (폰트 패밀리·크기)
- [ ] 버튼·입력 스타일 일관

### 사용성
- [ ] CTA가 명확하고 눈에 띄는가
- [ ] 레이블·설명이 행동 지향적인가
- [ ] 피드백(로딩·에러)이 있는가
- [ ] 취소·되돌리기가 가능한가

### 반응형
- [ ] 모바일 (≤640px) 동작
- [ ] 태블릿 (641~1024px)
- [ ] 데스크톱 (≥1025px)
- [ ] 터치 타깃 44px 이상

### 접근성
- [ ] 색 대비 WCAG AA (텍스트 4.5:1, 대형 텍스트 3:1)
- [ ] 색만으로 의미 전달하지 않음
- [ ] 포커스 표시 있음
- [ ] 이미지 alt 텍스트
- [ ] 키보드로 모든 인터랙션 가능

### 컨텐츠
- [ ] 카피가 명확·간결
- [ ] 전문용어 설명
- [ ] 플레이스홀더 vs 실제 카피 구분

### 상태
- [ ] 로딩 상태
- [ ] 빈 상태
- [ ] 에러 상태
- [ ] 성공 상태

## 출력

```markdown
# 디자인 리뷰: <대상>

**리뷰 일자**: YYYY-MM-DD
**대상**: <URL 또는 Figma>
**범위**: <전체 | 특정>
**리뷰어**: designer

---

## 1. 요약
- 전반 인상: <1~2문장>
- 발견 이슈: Critical N, High N, Medium N, Low N개

## 2. 이슈 목록

### [Critical] <이슈 제목>
- **영역**: 정보 위계 | 일관성 | 사용성 | 반응형 | 접근성 | 컨텐츠
- **현황**: <무엇이 문제인가>
- **영향**: <사용자에게 어떤 영향>
- **제안**: <구체적 개선안>

### [High] <이슈 제목>
...

### [Medium] <이슈 제목>
...

## 3. 체크리스트 요약
- 정보 위계: 3/3
- 일관성: 2/4 (미준수: 색·타이포)
- 사용성: 4/4
- 반응형: 2/3 (태블릿 미확인)
- 접근성: 3/5 (색 대비·키보드 필요)

## 4. 다음 단계
- [ ] Critical 이슈 1~2건 즉시 수정
- [ ] High 이슈는 다음 스프린트
- [ ] Medium 이하는 백로그
```

---

## 체크리스트

- [ ] 7개 관점 모두 리뷰
- [ ] 이슈마다 심각도 부여
- [ ] 이슈마다 구체적 개선안
- [ ] 요약 섹션에 수치 요약
- [ ] 다음 단계 명시

---

## 금지

- **취향 비판**: 개인 선호를 이슈로 처리 금지. 근거(가이드라인·데이터) 동반
- **개선안 없는 지적**: "이게 문제"만 말하고 해결책 없음 금지
- **일괄 수정 요구**: 모든 이슈를 동시 수정 요구 → 우선순위 매김
- **건설적 피드백 누락**: 잘된 점도 함께 언급

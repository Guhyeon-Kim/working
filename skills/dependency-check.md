# dependency-check

**버전**: v6.0
**주 사용 에이전트**: builder
**연계 스킬**: library-docs, error-handling

---

## 목적

새 의존성(라이브러리·패키지·외부 API) 추가 전 **의식적 점검**. 번들 크기·
라이선스·보안·유지보수 상태를 확인하여 기술 부채 사전 방지.

---

## 호출 시점

- `npm install <new-package>` 실행 전
- 기존 의존성 업데이트 (major 버전)
- 외부 API 통합 결정 전
- 프로젝트 의존성 감사 (주기적)

---

## 입력

- 대상 패키지·API 이름
- 용도 (왜 필요한가)

---

## 절차

### 1. 필요성 재검토
- 이미 쓰고 있는 것으로 해결 가능한가
  - lodash 추가 전: 네이티브 JS로 가능한가
  - 상태 관리 라이브러리 추가 전: React 내장으로 가능한가
- 직접 구현하면 얼마나 걸리는가 (적으면 자체 구현)
- 공식 문서가 이 라이브러리를 권장하는가

### 2. 라이브러리 건강도 점검

#### GitHub 지표
- **최근 커밋**: 3개월 내 활동
- **이슈·PR 응답**: 평균 응답 시간
- **Star·Fork**: 참고 지표 (절대 수치 무의미)
- **Contributors**: 1명 의존 → 위험
- **Release 주기**: 정기적·장기간 무소식

#### npm 지표
- **다운로드 수**: 주간 다운로드 (위클리)
- **버전 번호**: 0.x → 아직 불안정 신호
- **Last published**: 1년+ 방치 → 위험

### 3. 번들 크기 확인
- https://bundlephobia.com/ 에서 gzip 크기 확인
- 프론트엔드: 5KB 이하 권장, 10KB 이상 신중
- 대안 비교 (date-fns vs dayjs vs moment)

### 4. 라이선스 체크
- **허용**: MIT, Apache-2.0, BSD, ISC
- **주의**: GPL, AGPL (상업 프로덕트 주의)
- **금지**: 커스텀·불명

### 5. 보안 체크
- `npm audit` 결과
- Snyk·GitHub Advisory 확인
- 의존성 사슬의 취약점 (transitive deps)

### 6. 대안 비교
최소 2~3개 후보 비교:
| 라이브러리 | 주간 DL | 크기 | 최근 커밋 | 라이선스 | 메모 |
|---|---|---|---|---|---|
| A | 10M | 4KB | 2주 전 | MIT | 표준 |
| B | 500K | 2KB | 3개월 전 | MIT | 경량 |
| C | 50K | 8KB | 1년 전 | MIT | 비추 |

### 7. 결정·기록
- 선택 + 이유
- 기각한 대안 + 이유
- devlog `회고` 혹은 주석에 남김

## 금지

- **무검토 설치**: `npm i <pkg>` 바로 실행
- **major 버전 무검토 업그레이드**: Breaking change 체크 필수
- **GPL 도입 in 상업 프로젝트**: 라이선스 리스크
- **"Stars 많으니까 OK"**: Stars는 인기도 지표, 품질과 무관
- **의존성 폭주**: 비슷한 기능 3개 라이브러리 공존

## 예시

**상황**: 날짜 포맷팅이 필요하다.

**체크**:
- 자체 구현: `new Intl.DateTimeFormat(...)` 기본으로 해결 가능? → Yes, 단순 케이스
- 복잡한 케이스 (시간대·상대 시간): 라이브러리 고려
  - moment.js: 70KB (기각, 대형)
  - date-fns: 함수형 tree-shakable, 개별 import 작음
  - dayjs: 2KB, moment 호환 API
- 결정: 단순 포맷은 Intl, 복잡한 것만 dayjs 도입

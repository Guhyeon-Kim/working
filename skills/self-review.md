# self-review

**버전**: v6.0
**주 사용 에이전트**: tester, builder
**연계 스킬**: qa-test, review-request

---

## 목적

제출 전 본인의 산출물(코드·문서·디자인)을 **체크리스트 기반**으로 자가 검증.
"내가 만든 거니까 괜찮겠지" 편향 제거.

---

## 호출 시점

- builder: 구현 완료 후 tester에 넘기기 전
- tester: 테스트 리포트 제출 전
- designer: 와이어프레임·디자인 제안 전
- planner: 요건서 CEO 컨펌 요청 전

---

## 입력

- 제출할 산출물 (파일·문서·PR)

---

## 절차

### 공통 체크
모든 역할에 적용:

- [ ] 요청 범위와 실제 산출물이 일치하는가 (범위 이탈 없는가)
- [ ] 제목·요약이 내용을 정확히 반영하는가
- [ ] 인용·링크가 유효한가
- [ ] 오타·문법 확인
- [ ] 제3자가 읽을 때 필요한 맥락이 있는가

### 역할별 체크

#### builder 코드 자가 리뷰
- [ ] 요건서 수용 기준 각각이 구현됐는가
- [ ] TypeScript 타입 오류 없음
- [ ] ESLint 오류 없음 (warning은 허용)
- [ ] 콘솔 로그·디버깅 코드 제거
- [ ] 사용 안 하는 import·변수 제거
- [ ] 매직 넘버 상수화
- [ ] 외부 I/O에 에러 처리
- [ ] 민감 정보 하드코딩 X
- [ ] 기존 컨벤션 준수 (파일 구조·명명)
- [ ] 커밋 메시지 명확

#### tester 리포트 자가 리뷰
- [ ] 모든 수용 기준 검증 결과 있음
- [ ] BLOCK 판정 시 재현 경로 명확
- [ ] 발견 이슈에 심각도·원인 추정·수정 제안
- [ ] 판정(PASS/PASS_WITH_CONCERNS/BLOCK) 명시
- [ ] 테스트 코드 위치 명시
- [ ] 성능·보안 체크 포함 (해당 시)

#### designer 산출물 자가 리뷰
- [ ] 최소 2개 안 제시
- [ ] 모바일·데스크톱 양쪽
- [ ] 로딩·빈·에러 상태 포함
- [ ] 기존 디자인 시스템 준수
- [ ] 컴포넌트 매핑 (builder 힌트)
- [ ] 접근성 기본

#### planner 요건서 자가 리뷰
- [ ] 문제·사용자·성공 지표 명확
- [ ] Must/Should/Could 분류
- [ ] 수용 기준 측정 가능
- [ ] 엣지 케이스 5개 이상
- [ ] 비기능 요구사항 4가지
- [ ] 가정·리스크 명시
- [ ] CEO 컨펌 게이트 포함

### 한 걸음 물러서서 물어보기
- "3개월 뒤의 내가 이걸 이해할까?"
- "이걸 처음 보는 사람이 필요 질문 없이 작업 가능한가?"
- "내가 놓친 관점이 있을까?"

### 의도적 반대 관점
본인의 산출물을 **비판자 입장**에서 읽기:
- 어떤 가정이 틀릴 수 있는가
- 어떤 엣지 케이스가 빠졌나
- 어떤 부분이 모호한가

---

## 출력

### 자가 리뷰 체크 결과

```markdown
## Self-Review 체크 (YYYY-MM-DD)

### 공통
- [x] 범위 일치
- [x] 제목 정확
- [x] 링크 유효
- [x] 오타 없음
- [x] 맥락 충분

### 역할별 (builder)
- [x] 수용 기준 구현
- [x] 타입 오류 없음
- [ ] ⚠️ Rate limit 미구현 (builder 노트에 명시, tester 주의)
- [x] 로그·디버깅 제거
- [x] 컨벤션 준수

### 반대 관점 확인
- 비판: "매매 0개 상태에서의 UX는?"
  → 해결: 빈 상태 컴포넌트 추가함
- 비판: "1500개 매매에서 느려질 위험"
  → 인지함. 리스크 섹션에 기록

### 최종 상태
제출 가능 | 수정 필요 (<무엇>)
```

---

## 체크리스트

산출물 유형별 위 목록 참조.

---

## 금지

- **체크 생략**: 1개라도 체크 안 하면 누락
- **자기 옹호**: 불확실한 걸 "문제 없음"으로 처리
- **반대 관점 생략**: 비판적 시뮬레이션 안 하고 제출
